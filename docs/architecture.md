# MDF-Viewer Architecture

Strict MVC separation is mandatory. The prototype this project replaces failed
because data, UI, and plotting were tightly coupled; every decision here exists
to keep those concerns apart.

## Layers

| Layer | Package | May import | Must NOT import |
|-------|---------|------------|-----------------|
| Model | `mdf_viewer.model` | numpy, asammdf | PyQt6, pyqtgraph, view, controller |
| View | `mdf_viewer.view` | PyQt6, pyqtgraph | model-loading logic, asammdf |
| Controller | `mdf_viewer.controller` | model, view_model | — |
| View-model | `mdf_viewer.view_model` | model, PyQt6, pyqtgraph | — |

The **model** is pure data and never imports Qt. The **view** is pure UI and
holds no business logic. The **controller** coordinates the two and owns
cross-widget state. The **view-model** holds the few bridge objects that must
reference both a model object and its on-screen representation.

## Signal classes

- **`model.signal_data.SignalData`** — raw timestamps + sample arrays. Pure data.
- **`model.signal_metadata.SignalMetadata`** — name, unit, min/max, sample
  count, comment, and other MDF fields. Pure data, no samples.
- **`view_model.active_signal.ActiveSignal`** — a signal placed on the plot.
  Pairs `SignalData` + `SignalMetadata` with its PyQtGraph curve, ViewBox, and
  color. This is the Model↔View bridge, which is why it lives in `view_model`
  rather than `model` (it references Qt/PyQtGraph types).

## File I/O isolation

`model.mdf_loader.MdfLoader` is the **only** module that imports `asammdf`. It
translates a file on disk into the plain data classes above and raises
`MdfLoadError` on malformed or unreadable content, so the rest of the app never
touches the asammdf API and never crashes on bad input.

## Assembly

`app.run()` is the single place that constructs and wires the three layers
together. No layer constructs another's object graph.

## Glossary

A few terms that read as synonyms on a skim but mean different things (#138):

- **Engineering unit** — a signal's physical unit as recorded in the MDF file (V, °C, ...); used for mismatched-unit validation in `PlotStripe.merge_signals`/`sync_signals`.
- **Display unit** — one independent Y-autorange/swimlane target within a stripe: an ungrouped signal, a Merged group, or a Synced group (`PlotStripe._display_units()`). Not a physical unit — a grouping concept, unrelated to "engineering unit" despite the shared word.
- **Segment** — an Active Signals Table's per-stripe UI container, always 1:1 with a `PlotStripe` (`_add_segment`/`_segment_for_stripe`). A second name for a stripe's own AST-side representation, not a distinct structural concept — see "Per-Stripe Active Signals Table (#100)" below.
- **Stripe** — one `PlotStripe`: an independent plot pane with its own Y-axes, sharing the tab's one X-axis and cursor pair with every other stripe.

## Decision log

Decisions are recorded here when the "why" is not obvious from reading the code.
Grouped by topic; most-recent entries at the bottom of each group.

### Foundation

- **Qt binding:** PyQt6, licensed GPL-3.0-only by Riverbank (not LGPL — earlier misconception); PyQtGraph supports it.
- **Project license (1.x):** GPL-3.0-only (`LICENSE`, `pyproject.toml`), required by PyQt6's GPL terms for the combined work. Replaces the earlier `Proprietary` placeholder.
- **Build:** `pyproject.toml` (src-layout, entry point `mdf-viewer`) + `requirements.txt` / `requirements-dev.txt`.
- **`__version__`:** defined in `src/mdf_viewer/__init__.py` — the single source of truth. `pyproject.toml` uses `dynamic = ["version"]` with `[tool.setuptools.dynamic] version = {attr = "mdf_viewer.__version__"}` so it reads from `__init__.py` at install/build time rather than hardcoding a second copy. The installer `.iss` still requires a manual bump at release time (Inno Setup cannot read Python). Python code imports `__version__` directly rather than via `importlib.metadata` (editable installs cache stale metadata until reinstalled).

### Data model

- **`ActiveSignal` location:** `src/mdf_viewer/view_model/` (not `model/`), to keep the data layer free of Qt/PyQtGraph imports.
- **`ActiveSignal.__eq__ = object.__eq__`:** list `in` and `remove` also use `__eq__`, which the dataclass version raises on numpy arrays. Identity equality is set alongside `__hash__` so both dict and list operations are consistent.
- **`SignalMetadata.data_type` / `is_integer`:** `MdfLoader.load_signal` captures the raw asammdf dtype before the mandatory float64 conversion. `is_integer` is used by `PlotArea._SignalAxisItem` to suppress fractional ticks on discrete/integer signals. `data_type` (e.g. `"uint8"`) is displayed in `SignalInfoBox`.
- **Enum/string signal fallback:** `load_signal` retries with `raw=True` when physical values are non-numeric byte strings (common for CAN enum signals like gear position or state flags); raw integer encoding is numeric and plots correctly with the existing integer-tick axis.
- **`MdfLoadError` in `errors.py` (#46):** moved from `model/mdf_loader.py` so that `view/main_window.py` can import it without creating a view→model dependency. Both `mdf_loader.py` and `main_window.py` import from `mdf_viewer.errors`.
- **MDF4 `header.author`** does not round-trip via asammdf (stored in XML comment block). The `MeasurementInfo.extra` dict is available for raw fields if needed later.

### MVC wiring

- **View imports in controller:** `TYPE_CHECKING`-only — no runtime view imports; all views are injected as constructor parameters.
- **Protocol contracts in `controller/interfaces.py` (#50):** seven `typing.Protocol` classes declare every method each controller calls on its view dependencies. Interface segregation applied to `ActiveSignalsTable`: `SignalTableProtocol` (add/remove/clear rows — `AppController`) and `CursorValueSinkProtocol` (cursor value columns — `CursorController`) are separate. All protocols live under `TYPE_CHECKING` — no runtime cost.
- **MVC assembly:** `MainWindow` creates view widgets; `app.py` reads them to construct `AppController`; `set_controller` completes the wiring. No layer constructs another's object graph.
- **Signal color palette:** 8-color cycling tuple defined in `app_controller.py`; resets on `load_file()`.
- **Duplicate signal prevention:** `AppController.add_signal` checks `(group_index, channel_index)` against active signals' metadata before loading; returns `False` on duplicates (callers use the return value to count skips for the status bar message).
- **Single controller contact point for `MainWindow` (#48):** `MainWindow` no longer holds a reference to `CursorController`. All cursor actions (`toggle_cursor`, `press_cursor1`, `press_cursor2`, `zoom_to_cursors`, `set_cursor_mode_callback`) are proxy methods on `AppController`, which delegates to `_cursor_ctrl`. `set_controller(ctrl)` has no `cursor_ctrl` parameter.

### Plot area — per-stripe patterns (originally `PlotArea`, renamed `PlotStripe` by #97)

**Superseded class name, current content:** the #97 Plot Stripes refactor split the single `PlotArea` class into `PlotStripe`/`PlotStripesArea` (see "Plot Stripes (#97)" below) — every pattern in this section still applies today, just per-`PlotStripe` instance rather than to one app-wide plot area. Renamed in place below rather than left under the old class name, per #135.

- **`PlotStripe` multi-axis pattern:** `pi.vb` (main ViewBox) is the X-axis host only — no curves added to it. Each signal gets its own `ViewBox` with `setXLink(pi)` for shared X, and an `AxisItem('right')` placed at the next layout column. `pi.vb.sigResized` → `_update_view_geometries()` keeps extra ViewBoxes geometrically aligned.
- **`PlotStripe.zoom_to_fit`:** computes X bounds from `active.data.timestamps` across all signals, calls `pi.vb.setXRange` (propagates via XLink), then `vb.autoRange()` per signal for independent Y reset.
- **Curve downsampling:** each `PlotDataItem` in `PlotStripe.add_signal` has `setClipToView(True)` and `setDownsampling(auto=True, method="peak")`. The curve is constructed without data, added to its `ViewBox` via `vb.addItem(curve)`, and only then given data via `curve.setData(...)` — calling `setData` before the curve has a parent `ViewBox` made pyqtgraph fall back to the `PlotWidget` for `getViewBox()`, which raised `AttributeError: autoRangeEnabled` once downsampling was enabled.
- **Y-axis tick formatting:** `_SignalAxisItem` subclasses `pg.AxisItem`; float signals use `:.6g` (strips floating-point noise like "256.000000007"); integer signals snap ticks to integer positions and format as plain integers.
- **Drag-and-drop MIME types (`view/_mime.py`):** `SIGNAL_MIME_TYPE` (`application/x-mdf-viewer-signals`) carries the Signal Browser's drag payload — a JSON-encoded list of `(measurement_index, group_index, channel_index)` triples (not bare `[group_index, channel_index]` pairs — flattened to include `measurement_index` by #103, since one drag/multi-selection can span multiple measurements in the now-unified browser list). `ROW_MIME_TYPE` (`application/x-mdf-viewer-active-signal-move`, added by #116) separately carries an already-active signal being dragged between Active Signals Table segments/stripes — just each dragged `ActiveSignal`'s `id()`. Event filters installed on `_pw.viewport()` (`PlotStripe`) and `_table.viewport()` (`ActiveSignalsTable`) handle DragEnter/DragMove/Drop for both, without subclassing PyQtGraph or QTableWidget.

### Plot Stripes (#97)

- **Container structure:** each stripe is a separate `pg.PlotWidget`/`PlotItem` (not multiple rows in one shared `GraphicsLayoutWidget`), stacked in a `QSplitter(Qt.Vertical)`. Stripe-height resizing is the splitter's native drag-handle behavior rather than hand-built divider painting. Every stripe's own `pi.vb` continues to be X-linked across all stripes (not just within one), so the existing per-signal `ViewBox`/axis pattern (see "Plot area" above) is unchanged within a stripe — a stripe is just another `PlotArea`-like host.
- **Why not one shared `GraphicsLayoutWidget`:** a single shared scene would let one cursor `InfiniteLine` visually span every stripe without duplication, but PyQtGraph gives no splitter-equivalent for resizing `PlotItem` rows inside one layout — that would mean hand-rolling drag handles and manual `QGraphicsLayout` stretch-factor updates. Separate `PlotWidget`s get resize handles for free and keep each stripe's X-axis-tick visibility (only the bottom-most stripe shows ticks, REQ-PLOT-181) and active-stripe marker independent and easy to reason about.
- **Cursor lines vs. the delta-time line:** because each stripe is its own `QGraphicsScene`, a single `InfiniteLine` can't span stripes — cursor lines are duplicated one-per-stripe and kept in lockstep (REQ-PLOT-182). The delta-time line is different: it isn't tied to any curve's data, so it only needs to exist in the currently active stripe's `pi.vb` at a time, reparented (or recreated) on active-stripe change, with its vertical position remembered independently per stripe (REQ-PLOT-105).

### Cursor system

- **`CursorController` wiring:** optional dependency injected via `AppController.set_cursor_controller()`; all notify calls are guarded by `None` check so the cursor system can be omitted without touching `AppController`.
- **`CursorController` reads signals on demand (#49):** the controller receives a `get_active_signals: Callable[[], list]` at construction (`app.py` passes `lambda: controller.active_signals`). This eliminates a second authoritative list. `AppController.add_signal` calls `cursor_ctrl.refresh()` after appending; `remove_signal` calls `on_signal_removed` (label cleanup only, before ViewBox destruction) then `refresh()`.
- **CursorView lifetime:** `CursorView` is a `QObject` constructed in `app.py` after `MainWindow` so the PlotItem scene already exists. Tests keep the parent `PlotWidget` alive via a separate pytest fixture to prevent C++ object deletion.
- **Nearest-cursor label logic:** uses identity-based label keys `(cursor_index, active)` to avoid numpy `__eq__` ambiguity (same pitfall as `ActiveSignalsTable._find_row`).
- **Cursor label Y-tracking:** labels are added to the signal's own `ViewBox` so `setPos(x, y)` is in the signal's Y coordinate space and the label tracks Y pan/zoom automatically. `_labels` stores `(TextItem, ViewBox)` tuples. Cursor cleanup is called in `AppController` *before* `plot_area.remove_signal` so the ViewBox is still in the scene when `vb.removeItem(lbl)` runs.
- **Cursor labels show value only:** no unit suffix — the unit is already on the Y-axis and in the Signal Info Box.

### UI details

- **Identity-based row lookup in `ActiveSignalsTable`:** `ActiveSignal` is a mutable dataclass with numpy-array fields; `__eq__` raises `ValueError` on boolean coercion. All lookups use `is` via `_find_row()`.
- **`ActiveSignalsTable.remove_row` / `clear` ordering:** the table widget is mutated first (`removeRow` / `setRowCount(0)`), then `_signals` — `removeRow`/`setRowCount(0)` can synchronously emit `itemSelectionChanged` before returning, and the handler indexes into `_signals`. Mutating `_signals` first left it shorter than the row indices Qt reported, raising `IndexError`. `_on_selection_changed` also has a bounds check as a defensive fallback.
- **Recent files persistence:** plain JSON (no `QSettings`/registry) for transparency and portability; written immediately on successful load so a crash doesn't lose the entry; failed loads are never recorded; stale entries pruned silently when the File menu opens.
- **Recent files menu wiring:** `MainWindow` takes a provider callable (`settings.get_and_prune`) rather than a direct `Settings` reference, keeping the view layer free of settings knowledge; `File.aboutToShow` triggers the rebuild so the list is always fresh.
- **Signal Browser filter debounce (#9):** `_filter_edit.textChanged` (re)starts a single-shot `QTimer` (`_FILTER_DELAY_MS = 250`); `_apply_filter()` runs on timeout. Recursive filtering over a large channel tree is expensive so per-keystroke filtering felt sluggish. `populate()`/`clear()` stop the timer and apply the empty filter immediately.
- **Load busy feedback (#9):** `MainWindow._load_file()` is the single entry point for all three load paths. It sets a wait cursor and a persistent "Loading…" status message for the duration of `controller.load_file()`, restoring both in a `finally` block.
- **File drop confirmation:** `MainWindow._on_file_dropped` checks `controller.is_file_loaded` before replacing an open file; uses `QMessageBox.question` so the user can cancel.
- **Status bar skip notification:** `MainWindow._on_add_signals` counts duplicates via `add_signal`'s `bool` return value and calls `show_status` with a singular/plural message.
- **Toolbar icons:** custom PNGs in `src/mdf_viewer/resources/icons/`; 32×32 px 1× and 64×64 px `@2x` HiDPI variants; `_load_icon(name)` helper wires both sizes into one `QIcon`.
- **Theme-aware toolbar icons:** each icon has a light-gray variant (dark backgrounds) and a dark-gray `_light` variant (light backgrounds). `_icon_suffix()` reads `QApplication.styleHints().colorScheme()` once at startup. Detected once; no live theme-change handling.
- **Application icon:** `src/mdf_viewer/resources/icons/app_icon.ico`; set via `MainWindow.setWindowIcon()`, baked into the EXE via `icon=` in `mdf_viewer.spec`, and used as the installer icon via `SetupIconFile` in `mdf_viewer.iss`.
- **Windows taskbar icon (unfrozen run):** `app.py` calls `SetCurrentProcessExplicitAppUserModelID` on `sys.platform == "win32"` before creating the `QApplication`. Without it Windows shows `python.exe`'s icon in the taskbar when run via the debugger.
- **Splash screen:** `app.py` builds a `QSplashScreen` pixmap from `app_icon.ico` plus version text, shown before the heavier view/controller imports and closed via `splash.finish(window)`. Not unit-tested (consistent with the rest of `app.py`).
- **Help > About:** `MainWindow` has a `Help` menu with an "About MDF-Viewer" action that calls `QMessageBox.about()` with version, description, author, and GitHub link.

### License system

- **v2.0 direction:** v1.5 is the last free/GPL-3.0 feature release. v2.0 stays GPL-3.0 and adds an honor-based license-key system: a cosmetic "Licensed to: …" display with no hard enforcement. GPL does not prohibit charging money, only requires distributing source.
- **Feature-gating model (Option A):** paid features ship as normal GPL code gated by a license-key check; the gate can be removed by anyone building from source — acceptable under the honor model.
- **File format:** JSON with a `payload` block (fields inside are signed) and a `signature` field (base64 Ed25519). `format_version` lives inside the payload so it cannot be tampered with post-signing.
- **Canonical signing:** `json.dumps(payload, sort_keys=True, separators=(',', ':'))` — deterministic, no whitespace.
- **Tiers:** Personal (1 seat), Team (5 seats fixed, `TEAM_SEATS = 5`), Enterprise (0 = unlimited).
- **Perpetual + 2-year updates:** license never expires; `updates_until` date is signed into payload. After expiry the app keeps working but shows a notice.
- **Backwards compatibility:** new payload fields must always be optional with defaults — old licenses remain valid. Only key rotation invalidates old licenses (avoid rotating the key pair).
- **`load_stored()` never raises** — returns `None` on missing or corrupt file so a bad license file never prevents the app from starting.

### Plugin groundwork (#70)

- **`EventBus` is a separate `QObject` owned by `AppController`, not `AppController` itself:** keeps `AppController`'s own class hierarchy free of Qt inheritance, and lets #71 (`PluginContext`) hand plugins `context.events` — a small, fixed surface — rather than a reference to all of `AppController`.
- **Event payloads are per-event dataclasses (`FileLoadedEvent`, `SignalAddedEvent`, ...) carried via `pyqtSignal(object)`, not positional typed args:** upcoming multi-measurement/stripe work (#100/#101/#106) will want to add fields like `measurement_id`/`stripe_id` to these events later. A dataclass field addition doesn't require any existing `.connect()` callback to change; a positional signal signature only stays compatible if new args are strictly appended and never inserted — the dataclass removes that footgun instead of just documenting around it.
- **Purely additive, not a rewiring of internal coordination:** `AppController` keeps every direct call it already makes to `_cursor_ctrl`/`_table`/`_plot`; it also emits an event alongside. No existing internal module was migrated to subscribe instead of being called directly — that direct-call structure is already well-tested and explicit, and there's no current subscriber to justify the swap.
- **`cursor_moved` wiring uses callback injection, not a Qt signal on `CursorController` itself:** `CursorController.set_position_changed_callback()` matches the existing `set_mode_changed_callback` pattern, keeping `CursorController` testable without a `QApplication` and decoupled from `AppController` (its own stated design goal). `AppController.set_cursor_controller()` wires the callback to `self.events.cursor_moved.emit(...)`. Fires on drag, chevron-fetch, and arrow-key step, and once from `_commit_mode` (covers initial cursor placement and `restore()`); suppressed while `HIDDEN`.
- **Help menu order:** License Key action first, separator, About last.
- **License state read once at startup; restart required after import (#51):** `app.py` calls `license_manager.load_stored()` once; `window.set_license(info, manager)` is called once and never again. `MainWindow._on_license()` opens the dialog and does nothing on accept — the dialog's own "restart required" message is the canonical flow.
- **Repo/secrets:** the Ed25519 private key and `generate_license.py` are saved at `C:\Users\andal\Documents\mdf-viewer-private\` — never commit either to any repo. `.gitignore` blocks `generate_license.py`, `*.lic`, and `*.pem`.

### Info/Properties Side Drawer (#98)

- **Shared `DockablePanel` helper extracted from the left-panel pin/hover code:** `MainWindow`'s existing pin/hover/slide mechanism (`_pinned`, `_drawer_shown`, `_hover_timer`, `_panel_anim`, `_toggle_pin`, `_slide_panel`, `_check_hover`, and the drawer-mode branch of `_update_child_geometries`) is pulled out into a reusable `DockablePanel` class, parameterized by which edge it docks to (left/right), rather than duplicating that geometry/animation logic mirrored for the right edge. `MainWindow` composes two `DockablePanel` instances — one for the existing left panel (Signal Browser/Measurement Info), one for the new right-side Info/Properties drawer — instead of hand-rolling the same pin/hover math twice. This is a refactor of already-working, tested code; it should land as its own step before the drawer-specific behavior is added, so a regression is traceable to "the extraction" vs. "the new feature."
- **Third `_content_splitter` column, not nested in `_right_splitter`:** the drawer becomes a sibling of `plot_area` and the Active Signals Table in `_content_splitter` (`plot_area | active_signals_table | info_drawer`), giving it full content-area height independent of the AST's height. `_right_splitter` loses `signal_info_box` and ends up holding only `active_signals_table` — worth revisiting during implementation whether `_right_splitter` should be collapsed away entirely (a `QSplitter` with one child is unnecessary) or kept for a future second widget in that column.
- **`SignalInfoBox` internal layout: vertical stack replaces `QTabWidget`.** `_tabs` is removed; the Info section and `_props_widget` (Properties) are placed directly in a `QSplitter(Qt.Vertical)`, each preceded by a small `QLabel` header ("Info" / "Properties") for identity now that tab labels no longer provide it. Today's `_tabs.setTabEnabled(1, False)` (Properties disabled when nothing is selected) becomes `_props_widget.setEnabled(False)`. This inner splitter's sizes are persisted the same way as the outer ones.
- **Persistence:** `_capture_splitter_sizes()` / `_apply_splitter_sizes()` gain a new `"info_drawer"` key mirroring today's `"left_panel"` entry (`pinned`, `width`), plus the Info/Properties inner-splitter sizes — both stored in the same `.mvc` splitter-sizes dict, no new top-level session-file section needed.
- **Scope boundary with #100:** `SignalInfoBox` stays a single `MainWindow`-owned instance, not duplicated per stripe. It reacts to whichever signal was most recently selected across any stripe's Active Signals Table. #100 (Per-Stripe Active Signal Table) may need to revisit this if per-stripe info display is later required — out of scope for #98.

### Main Widget Tabs (#99)

- **`AppController` stays a single instance, made internally tab-aware, rather than one `AppController` per tab.** A new controller-side `TabWorkspace` bundle wraps everything a tab's requirements say must be independent: the per-tab active-signal list, its `PlotStripesArea` reference, its own `ZoomController`, and its own `CursorController` + `CursorStripesView`. `AppController` holds `_workspaces: list[TabWorkspace]` + `_active_tab_index`; its existing public API (`add_signal`, `remove_signal`, zoom/cursor actions) delegates to `self._workspaces[self._active_tab_index]` instead of flat instance attributes. This mirrors the pattern already proven for stripes — one `CursorController` composing one `CursorView` per stripe (see "Cursor system" above) — promoted one level up to tabs.
- **Why not one `AppController` per tab:** duplicating `AppController` would also duplicate `EventBus` ownership (`self.events = EventBus()` is currently constructed inside `AppController.__init__`), or require retrofitting constructor-injection to share one bus across instances. A router layer on top of N controllers adds a whole new coordination surface for no benefit over making the one controller tab-aware internally.
- **One shared `EventBus` for the whole app, not one per tab.** Matters for the plugin architecture (#71–#76, milestone 2.3): plugins subscribe once and shouldn't need to know tabs exist. Event dataclasses gain an optional `tab` field alongside the `stripe` field `SignalAddedEvent` already has, using the extensibility the #70 design note anticipated (`events.py:8-10`).
- **Shared panels are unaffected.** Signal Browser, Measurement Info Box, and the Info/Properties drawer (#98) keep calling the one `AppController` exactly as they do today — no new routing/protocol surface needed for them, since tab-awareness is entirely internal to `AppController`.
- **`MainWindow` grows a `QTabWidget`** in place of today's single `plot_area`/`active_signals_table` pair inside `_content_splitter`. Each tab page is a plain view-only widget pairing a fresh `PlotStripesArea` + `ActiveSignalsTable` — no business logic in the pairing itself, matching the MVC rule. Creating a tab: `MainWindow` builds the new view pair via a factory method, then calls `AppController.create_tab(views)` to register the matching `TabWorkspace` — the same "MainWindow builds views, controller wires them" assembly pattern `app.py` already uses at startup, just invoked again at runtime instead of only once.
- **Tab-bar mechanics map onto `QTabWidget` largely for free:** `setTabsClosable(True)` + `tabCloseRequested` (routed through a confirmation check against that tab's `TabWorkspace` signal count, mirroring the stripe-deletion warning), `setMovable(True)` for drag-to-reorder, a corner-widget "+" button for new-tab creation, and `QShortcut`s for `Ctrl+Tab`/`Ctrl+Shift+Tab` cycling (`QTabWidget` has no built-in cycling shortcut).
- **Empty-state placeholder (0 tabs):** a small `QStackedWidget` wraps the `QTabWidget`, switching to a placeholder page with a "New Tab" action when the last tab is closed, rather than `QTabWidget` being left in an unsupported zero-tab state.
- **Implementation notes (M1–M6):**
  - `current_workspace` is a public `AppController` property (not `_current`), so `app.py`'s tab-wiring factory can attach a freshly built `CursorController`/`ZoomController` to a specific tab — including the first one, created by `__init__` — without `AppController` importing those concrete classes itself.
  - `app.py`'s original one-time wiring block (build `CursorStripesView`, wire stripe signals, build `CursorController`, `set_cursor_controller`, wire `active_stripe_changed`, build `ZoomController`, `set_zoom_controller`) was extracted into `_wire_tab(controller, workspace, settings)`, reused for the first tab at startup and for every runtime-created tab via a new `_add_new_tab()` factory, injected into `MainWindow` as `window.set_tab_factory(...)` — the same callback-injection pattern already used for `set_recent_files_provider`.
  - `MainWindow._wire_tab_view(plot_area, active_signals_table)` extracts the per-tab signal/slot connections out of `set_controller()` (everything that touches `plot_area`/`active_signals_table`); the `signal_info_box`/`signal_browser` connections stay in `set_controller()` since those panels are shared, not per-tab.
  - Each tab page is a `QSplitter(Horizontal)` pairing `plot_area` + `active_signals_table` (built by `MainWindow._make_tab_page`), with `.plot_area`/`.active_signals_table` stashed as plain Python attributes on the splitter instance for later lookup (`_all_active_signals_tables()`, close-tab handling) — simpler than introducing a dedicated tab-page class.
  - **"Does this tab have signals" is answered by the controller, not the view:** `AppController.tab_has_signals(index)` checks `self._workspaces[index].active` — `ActiveSignalsTable` doesn't expose a public row-count API, and checking via the controller keeps "what counts as having signals" out of the view layer.
  - **`QTabBar`'s default `selectionBehaviorOnRemove` is `SelectRightTab`** (confirmed empirically, not just from docs) — the opposite of REQ-PLOT-253's left-neighbor rule. `MainWindow._on_tab_close_requested` computes `new_index = max(0, index - 1)` *before* calling `removeTab`, then calls `setCurrentIndex(new_index)` explicitly afterward.
  - **Test gotcha:** a `MagicMock`-based controller's `tab_has_signals(index)` call returns a truthy `MagicMock` by default (`MagicMock.__bool__` is `True`), which silently triggers a real, unmocked `QMessageBox.question()` in tests that don't set `mock_controller.tab_has_signals.return_value` explicitly. In this environment such a dialog doesn't hang forever (something auto-dismisses it after ~5s) but tests should never rely on that — always set the return value explicitly and, for the warning path itself, patch `QMessageBox.question`.

### Multiple Measurements (#101)

- **`LoadedMeasurement` lives in `model/`, not `controller/`.** A new `model.loaded_measurement.LoadedMeasurement` dataclass bundles an `MdfLoader` instance, its `MeasurementInfo`, a user-facing `label: str` (file-derived, disambiguated on collision — REQ-FILE-027), and a mutable `offset_s: float` (REQ-FILE-026/REQ-PLOT-304). Nothing here needs Qt — same rationale that already puts `MdfLoader` itself in `model/` despite being stateful/orchestrating I/O. `AppController` holds `self._measurements: list[LoadedMeasurement]` as a single global pool shared across every tab, mirroring the already-shared Signal Browser (REQ-FILE decision: "global pool, shared across tabs"), replacing today's single `self._loader`.
- **Per-measurement offset is applied at render time via `ActiveSignal.display_timestamps`, not via a second ViewBox/X-link per measurement.** The requirement that X-*zoom* always stays in lockstep across every measurement (REQ-PLOT-302), with only X-*pan* independent per measurement (REQ-PLOT-303), means the existing single-shared-ViewBox X-link architecture (`PlotStripe._pi.vb`, `PlotStripesArea._on_stripe_x_changed`'s manual one-to-many broadcast) needs no structural change at all — it already only ever moves in lockstep. Instead, `ActiveSignal` (view_model) gains a `measurement: LoadedMeasurement | None` field — a live *reference*, not a copied offset value, per the single-source-of-truth lesson from #100 (`feedback_single_source_of_truth`) — and a computed `display_timestamps` property (`data.timestamps + measurement.offset_s`, or raw `data.timestamps` unchanged when `measurement is None`, e.g. in tests that construct `ActiveSignal` directly). Every place in `PlotStripe`/`PlotStripesArea` that reads `active.data.timestamps` for X-positioning — `add_signal`'s `curve.setData`, `set_step_mode`, `zoom_to_fit`'s `t_min`/`t_max`, `swimlanes`'/`zoom_y_to_view`'s visible-range mask — switches to `display_timestamps`. `model.interpolate.interpolate()` symmetrically subtracts `active.measurement.offset_s` from the cursor's display-time `x` before indexing into the signal's raw timestamps (REQ-PLOT-305).
- **Per-measurement axis rows are new `AxisItem`s stacked below the existing "Time" axis, all linked to the same shared ViewBox — not separate ViewBoxes.** Each row's `tickStrings` subtracts that row's own measurement offset before formatting, so the row displays that measurement's *real recorded time* under one shared ruler/range (REQ-PLOT-300). Built only in the bottom-most stripe via a new `PlotStripe.set_measurement_axes(measurements)`, mirroring the existing bottom-most-only rule already used for `set_show_x_axis_ticks` (REQ-PLOT-301); every other stripe is passed an empty list. A new `_MeasurementAxisItem` (same subclassing pattern as `_SignalAxisItem`) overrides `mouseDragEvent` to translate a horizontal drag into an `offset_s` delta (via the ViewBox's `viewPixelSize()`) instead of forwarding to the ViewBox's own pan handling — critically, it must *not* touch the shared ViewBox range itself, since "drag the interior = pan everything together" vs. "drag one measurement's own row = pan that measurement alone" are REQ-PLOT-303's two distinct gestures. A drag fires a callback that updates `LoadedMeasurement.offset_s` and recomputes `display_timestamps`/`curve.setData(...)` for every active signal belonging to that measurement, across every stripe *and every tab* — a direct consequence of the global-pool decision above. Continuous `setData()` calls during a drag are a watch-item for large files; debounce like `ZoomController`'s gesture-settle pattern if live-testing shows it's needed, but don't add it speculatively.
- **Channel identity becomes measurement-scoped.** `group_index`/`channel_index` are only unique within one file, so `AppController.add_signal`'s existing duplicate-check (REQ-PLOT-020) and the Signal-Browser-to-plot drag-drop MIME payload (`SIGNAL_MIME_TYPE`, today a bare JSON list of `[group_index, channel_index]` pairs, see `_mime.py`) both need the owning `LoadedMeasurement` to disambiguate. `AppController.add_signal` gains a leading `measurement: LoadedMeasurement` parameter. The MIME payload becomes a JSON object `{"measurement_index": i, "items": [[gi, ci], ...]}` rather than a bare list — one measurement per drag gesture, since a drag always originates from whichever single measurement's tree the Signal Browser currently shows. `SignalBrowser.add_signals_requested` likewise needs the browser's currently-selected measurement threaded through to the `AppController.add_signal` call site in `MainWindow`.
- **Signal Browser gets a minimal measurement selector**, not full cross-measurement browsing (that's #103). A combo box (or small strip) above the tree swaps which `LoadedMeasurement.channel_tree()` is populated into the existing tree widget; hidden entirely with exactly one measurement loaded (REQ-BROWSER-050/051).
- **Active-signal display-name prefixing (REQ-PLOT-306) is applied at the formatting layer, not baked into `SignalMetadata.name`.** It's read from `active.measurement.label` wherever a signal's display name is rendered (`ActiveSignalsTable`, `SignalInfoBox`) — the same layer `refresh_display_names()`'s shortening rule already occupies — keeping the underlying channel name used for by-name resolution (REQ-FILE-031, near-match, `.mvc` restore) unaffected, per REQ-PLOT-307. Cursor value labels are untouched by this: `cursors.py`'s `update_labels` only ever renders a bare value or enum label (see "Cursor system" above, "Cursor labels show value only"), never a signal name, so there is no cursor-label prefixing to build.
- **`AppController.load_file` splits into `replace_measurements(paths)` and `add_measurements(paths)`.** `replace_measurements` generalizes today's single-file `load_file` semantics to N files (REQ-FILE-021, REQ-PLOT-260's per-tab re-resolution unchanged). `add_measurements` is purely additive — appends new `LoadedMeasurement`s, touches no existing tab's signals/zoom/cursor/undo state (REQ-FILE-022, REQ-PLOT-261). Both collect per-file `MdfLoadError`s into one summary rather than surfacing the first failure only (REQ-FILE-023/024) — a different pattern from `MainWindow._on_add_signals`'s today per-channel loop, since a load failure here is file-level, not channel-level. `MainWindow` gains the Replace-vs-Add `QMessageBox.question` prompt (REQ-FILE-020) wherever `controller.measurement_count > 0`, in both `_on_load_file` and `_on_file_dropped`.
- **`AppController.close_measurement(measurement)`** walks every tab's `active` list, removing (via the existing full `remove_signal` pipeline — cursor labels, table row, plot ViewBox) every `ActiveSignal` whose `.measurement is measurement`, drops it from `self._measurements`, and rebuilds the bottom-most stripe's axis-row stack in every tab. Mirrors the stripe/tab close-with-active-signals warning (REQ-FILE-028).
- **`.mvc` persistence is untouched by #101, deliberately.** `capture_config`/`restore_config` stay scoped to `current_workspace` and a single `measurement_path` field, matching the provisional single-measurement scope #99 already left for `.mvc` (see "Main Widget Tabs" above); multi-measurement session save/restore is #106's job, not #101's.

- **`MdfLoader.find_similar_signal_by_name(name)`** (model layer, alongside the existing `find_signal_by_name`): same `channel_tree()` traversal, but matches on the substring before each name's own last `\` rather than the full name, excluding exact matches and any name with no `\` at all (REQ-FILE-032/033). `AppController.find_similar_signal_by_name(name)` is a thin proxy, mirroring the existing `find_signal_by_name` proxy.

### Per-Stripe Active Signals Table (#100)

- **`ActiveSignalsTable` structure:** a shared header-only `QTableWidget` (never given rows — REQ-PLOT-272) above a `QSplitter(Vertical)` of per-stripe "segments," each segment being a small container widget stacking `[name_label, _ActiveTable]`, plus the existing Remove Signal/Remove All button row. `_configure_columns()` applies identical column setup to the header and every segment; the header's `sectionResized` pushes new widths to every segment one-directionally (segment headers are hidden/inert, so no re-entrancy guard needed there).
- **Data model — one shared list, not one per segment:** `ActiveSignalsTable._signals: list[ActiveSignal]` (mirrors `TabWorkspace.active`) plus `_stripe_for_signal: dict[ActiveSignal, stripe]` (mirrors `PlotStripesArea._signal_stripe`) are the only state; a segment's rows are always a pure rendering of `_segment_signals(seg)` — signals whose stripe matches, filtered from the shared list, preserving its order. No segment independently tracks "its own" signals. **Why:** the first cut gave each segment its own `signals` list, kept in sync by hand; a gap in that sync (moving a signal to a different stripe via "Move to Stripe" never told the AST, only the plot's `_signal_stripe`) orphaned a signal's row when its old stripe was later deleted. Deriving segment contents from one shared list instead of N parallel copies makes that bug class structurally impossible — see `feedback_single_source_of_truth` in project memory. `ActiveSignalsTable.move_to_stripe(actives, stripe)` (new) just repoints `_stripe_for_signal` entries and re-renders the affected segments; `AppController.move_signals_to_stripe`/`move_signals_to_new_stripe` call it alongside the existing plot-side call.
- **Divider alignment is a constant-offset subtraction, not a ratio.** `PlotStripesArea`'s stripe splitter and the AST's segment splitter never have equal total heights (the AST's sits below a header and above a button row) — forwarding sizes between them via fractions/ratios, or via `QSplitter.setSizes()`'s own squash-to-fit, only preserves *proportions*, not equal absolute pixel heights, so a middle segment in a 3+ way split visibly drifted from its stripe the further a divider was dragged. The fix: `ActiveSignalsTable` computes fixed `_top_size_offset`/`_bottom_size_offset` once (header height + spacing; button row height + spacing) and subtracts them from only the *first*/*last* segment's target size when applying `PlotStripesArea`'s absolute stripe sizes (and adds them back when reporting segment sizes upward) — every *interior* segment then matches its stripe's height exactly, with zero rounding tolerance needed. Also: a freshly created/bootstrapped segment must be pushed its stripe's real size immediately (`PlotStripesArea.create_stripe()`/`delete_stripe()` now emit `stripe_sizes_changed` themselves, and `MainWindow._wire_tab_view` pushes the bootstrap size once) — without this, a new segment silently kept whatever arbitrary size `QSplitter.addWidget()` gave it until a drag happened to touch that exact divider. This was root-caused via temporary `print()` diagnostics on the real sync code, not by theorizing — same pattern as the #78 postmortem.
- **Segment name label: a dedicated layout row, not an overlay.** An early version floated the label over the segment's top-left corner (positioned via `setGeometry`, not a layout) specifically to avoid touching the divider-alignment math above. Live-tested and found to cover part of the first signal row. Moved into a small per-segment container (`[QLabel, _ActiveTable]` in a `QVBoxLayout`) instead — this needed **no** change to the alignment math, since REQ-PLOT-273/274 only constrain each *container's* total height to match its stripe's, not how a container subdivides itself internally (the label is just less space left over for data rows, the same way the shared header/button chrome already is for the first/last segment).
- **Cross-segment selection defers "clear sibling segments" to mouse release, not press.** Qt's own `QAbstractItemView` already avoids collapsing an existing multi-selection when you press (without a modifier) on an already-selected row, specifically so a press-and-drag on it drags the whole group — deferred to release only if no drag happens. The first cut's cross-segment "plain click clears other segments" logic fired unconditionally on every press, pre-empting that native deferral and breaking drag of an existing cross-segment (Ctrl-built) selection. Fixed by checking whether the press landed on an already-selected row: if so, defer clearing to `_ActiveTable.on_click_release`, which only fires when `mouseMoveEvent` never crossed the drag-distance threshold this press. See `feedback_deferred_selection_collapse` in project memory.
- **Row drag-to-move fully replaces Qt's native `InternalMove`** with one custom `QDrag` + MIME mechanism (`_ROW_MIME_TYPE`, JSON list of `id(ActiveSignal)` — valid only within the process's own lifetime, which a drag never outlives), driven manually via `_ActiveTable.mouseMoveEvent` (not `setDragEnabled`, which would fight with it) and handled through the same eventFilter `DragEnter`/`DragMove`/`Drop` pattern already used for `SIGNAL_MIME_TYPE` drops from the Signal Browser. Same-segment reorder and cross-segment move share one code path (`_apply_row_move`): same-segment is just the case where every dragged signal is already in the target's stripe. The drag payload is always the full cross-segment selection (`_selected_signals()`), not just the pressed segment's own rows, so dragging one row of an existing multi-segment selection moves the whole group.
- **Stripe naming:** `PlotStripe.name: str` is a plain mutable attribute (matches `ActiveSignal`'s own mutable display fields); the creation-order counter (`"Stripe N"`) lives on `PlotStripesArea`, scoped per tab, mirroring `MainWindow._tab_counter`'s ownership pattern but never shared across tabs. Renaming (`ActiveSignalsTable._rename_segment`, double-click the label) mirrors `MainWindow._rename_tab`'s direct `QInputDialog` pattern exactly — no controller round-trip, since stripe names (like tab names) aren't persisted/model state yet (REQ-PLOT-294, deferred to #106).

### PyQtGraph/Qt Teardown Pitfall — orphaned-but-alive graphics items (native crash class) (#120)

**The pitfall:** `GraphicsLayout.removeItem(item)` / `QGraphicsScene.removeItem(item)` / `item.hide()` only detach a `QGraphicsItem` from layout positioning, the scene, or visibility — none of them destroy the underlying Qt object. Without an explicit `item.deleteLater()` afterward, the item stays alive in memory, still fully wired into whatever signal/slot connections it had (range-sync, view-linking, etc.), just invisible and unlaid-out. It looks correctly "removed" in every way a quick visual check would catch.

**Why it's dangerous, not just wasteful:** an orphaned-but-alive `ViewBox`/`AxisItem`/curve can still react to events fired by *other*, still-live objects it remains connected to (e.g. `sync_x_range()` keeps every stripe's X `ViewBox` in sync, so an X-range change anywhere fires `sigRangeChanged` on every stripe's `ViewBox`, including ones whose axis/spacer was "removed" this way). Separately, `ViewBox.removeItem(curve)` internally does `scene.removeItem()` then `setParentItem(None)` — the None-parent state makes PyQtGraph's `getViewBox()` wrongly cache the enclosing `QGraphicsView` as the item's view, which then crashes the next time `viewRangeChanged()` calls `view.autoRangeEnabled()` on it (see `_reparent_curve`'s docstring in `plot_stripe.py`). Both failure modes manifest as a genuine native `EXCEPTION_ACCESS_VIOLATION` (segfault) with **zero Python traceback** — the crash happens below the interpreter, in Qt's own C++ code, often a long time (and several unrelated operations) after the actual leak was created, which is what makes it so hard to trace back.

**Found this way three times in `plot_stripe.py` alone** (per-signal `ViewBox`/`AxisItem` in `_destroy_vb_and_axis`, the per-measurement X-axis row in `set_measurement_axes`, the axis-alignment spacer in `set_axis_padding`) plus once in tab-close (`AppController.remove_tab` dropped a `TabWorkspace` without running its still-active signals through `remove_signal` first, and `MainWindow._on_tab_close_requested` never `deleteLater()`'d the closed tab's page widget at all). All were root-caused not by reading the code and spotting the bug directly, but by **matching Windows Error Reporting crash dumps** (`%LOCALAPPDATA%\CrashDumps\python.exe.<pid>.dmp`, parsed with the pure-Python `minidump` package — no WinDbg/cdb available on this machine) against `print()`-instrumented reproductions to find the exact last line of Python code that ran before each crash, then reasoning backward from there. A segfault with no traceback will not show up in pytest, in a debugger's normal exception view, or in `try`/`except` — only a live repro plus either a crash dump or synchronous print-before/print-after instrumentation narrows it down.

**What to check when adding/reviewing any code that tears down a `ViewBox`, `AxisItem`, curve, or other PyQtGraph/`QGraphicsItem`:** does the teardown path end in `deleteLater()` (after removing from its scene, if it has one), or does it stop at `removeItem()`/`hide()`? The same question applies whenever a widget or its children get detached from a container they were added to (`QTabWidget.removeTab()`, `QSplitter`, `QLayout.removeWidget()`, etc.) — none of Qt's own "remove" APIs imply "destroy."

**Found a fourth instance in `view/cursors.py`, fixed as #134 (2026-07-12)** — a whole-codebase architecture review audited every teardown site against this checklist and found `CursorStripesView`'s four cursor-value-label teardown sites (`apply_mode`'s stale-ViewBox replacement, its stale-key cleanup, `remove_labels_for`, `clear_labels`) all called `vb.removeItem(lbl)` on a `pg.TextItem` with no follow-up `deleteLater()` — never audited when #120 landed, since #120's own fix only touched `plot_stripe.py`/`plot_stripes_area.py`. Not confirmed to have crashed in practice (an orphaned label isn't as reliably re-triggered afterward as a live-linked curve/ViewBox), so this was a latent risk, not an active one. Fixed with a new `cursors.py`-local `_destroy_label(lbl, vb)` helper mirroring `PlotStripe._destroy_curve`'s exact pattern — deliberately skips `ViewBox.removeItem()` entirely (manually pops the item from `vb.addedItems`/clears `_itemBoundsCache`, then calls `deleteLater()` directly) rather than just appending a `deleteLater()` call after the existing `removeItem()`, since `removeItem()` itself is what triggers the dangerous `setParentItem(None)` state — the same reasoning `_destroy_curve`'s own docstring already documents. Kept as a small local duplicate of `_destroy_curve`'s logic rather than a new cross-module shared utility, since it's two call sites total and the project's convention favors avoiding premature abstraction for that scale.

### Measurement Synchronization (#102)

- **"Synchronize" is a display-only change, reusing the existing per-measurement-row machinery instead of adding a new rendering path.** `PlotStripe.set_measurement_axes(measurements)` already builds exactly one `_MeasurementAxisItem` row per entry in whatever list it's given. Synchronized mode is simply `PlotStripesArea._update_x_axis_tick_visibility()` passing `[self._measurements[0]]` (the reference — always the first-loaded measurement, REQ-PLOT-312/313) instead of the full pool when `self._synchronized` is set, instead of the full `self._measurements` list. No offset, `display_timestamps`, or cursor-lookup math changes (REQ-PLOT-311) — same reasoning as #101's existing offset/display split.
- **`_MeasurementAxisItem` gains a `draggable: bool = True` constructor param**, guarding the top of `mouseDragEvent`. Needed because that method unconditionally mutates `measurement.offset_s` on drag regardless of whether an `on_offset_changed` callback is passed — reusing it unguarded for the synchronized single row would let the user silently drag the reference's offset while "synchronized," contradicting REQ-PLOT-314 (Un-sync is the only way to re-adjust). `set_measurement_axes` passes `draggable=False` for every row it builds while synchronized (trivially just the one reference row).
- **Synchronized state is global, not per-tab** — `AppController.self._measurements_synchronized: bool`, alongside the already-global `self._measurements` pool (#101's "global pool, shared across tabs" decision extends naturally here; REQ-PLOT-313's all-or-nothing rule wouldn't make sense scoped per-tab when the same measurements are shared across every tab). `toggle_measurements_synchronized()` flips it and fans out to every tab exactly like `_refresh_measurement_axes()` already does for the measurement pool itself — `PlotStripesArea.refresh_measurement_axes(measurements, synchronized)` gains the flag as a second parameter, stores it, and re-runs `_update_x_axis_tick_visibility()`.
- **The corner Synchronize/Un-sync button is new UI plumbing, not a reuse of an existing pattern**: a small overlay button parented to `self._pw` (the bottom-most stripe's `PlotWidget`/`QGraphicsView`), pinned to a corner via a `resizeEvent` override rather than participating in `PlotStripe`'s outer `QHBoxLayout` (unlike `_active_marker`, which sits *beside* the plot, this needs to sit *on top of* it, near the measurement axis rows). Shown/hidden and relabeled by a new bottom-stripe-only method mirroring `set_measurement_axes`'s/`set_show_x_axis_ticks`'s own bottom-only convention — **hidden entirely** (not shown-disabled) whenever fewer than two measurements are loaded, matching how the whole measurement-axis-row area doesn't exist yet with 0 measurements today. The Edit menu action mirrors it 1:1 but stays visible-and-disabled instead of hidden (standard menu convention), and both call the same `AppController.toggle_measurements_synchronized()`.
- **Not persisted.** `.mvc` save/restore is untouched by this issue, matching #101's own explicit deferral of multi-measurement persistence to #106 — `self._measurements_synchronized` resets to `False` on every fresh load, same as every `LoadedMeasurement.offset_s` already does.
- **Condition-based alignment ("align at the moment Signal X reaches Value Y") is out of scope**, split into #121 and deferred until #110 (Signal Value Search) exists to reuse rather than duplicate.

### Signal Browser Multi-Measurement Unification (#103)

- **`LoadedMeasurement.label` is repurposed in place as the user-facing "short name," not renamed or duplicated into a second field.** Same attribute, same ~10 existing read sites (AST/SignalInfoBox prefixing, axis-row labels — REQ-PLOT-300/306). Only its generation and mutability change: `make_label()` defaults to `"M1"`, `"M2"`, ... by load-order position instead of the file stem (REQ-FILE-027), and the field becomes user-editable (`AppController.rename_measurement(measurement, new_name) -> bool`) instead of write-once-at-load. The view reverts the edit box's text on a `False` return (duplicate) rather than blocking keystrokes.
- **The load-order position fed to `make_label()` is `AppController._measurement_load_counter`, a monotonic counter — not `len(self._measurements)` or `len(labels)` computed from whatever's currently loaded.** Found live-testing: with the count-based approach, loading 3 (M1-M3), renaming M1, closing M1 and M2 (leaving only M3), then adding 2 more reused the now-empty positions — `len(...)` restarted from 1, producing "M2" then colliding with the still-loaded "M3" (disambiguated to "M3 (2)"), even though M1/M2 were themselves long gone. The counter only increments (once per measurement actually created, in `replace_measurements`/`add_measurements`/`load_file`) and is reset to 0 only by a fresh Replace or the legacy `load_file` — the same "starts over" point that already resets `offset_s`/Synchronized/Primary — so `add_measurements` after closing measurements always continues from the next never-yet-used name instead of racing the current pool's size.
- **Primary measurement is one `AppController._primary_measurement: LoadedMeasurement | None` reference, not a per-`LoadedMeasurement` bool field.** A bool field risks the same desync class the #100 single-source-of-truth lesson already burned once (two `True`s, or zero) — a single reference makes "exactly one Primary" structural rather than something that has to stay in sync. Auto-set to the first-loaded measurement on `replace_measurements`/initial `load_file` (REQ-PLOT-322); unaffected by `add_measurements` (REQ-PLOT-322); on `close_measurement`, if the removed measurement was Primary, reassign to `self._measurements[0]` **of whatever remains, guarded for the empty case** (`self._primary_measurement = self._measurements[0] if self._measurements else None`) — `self._measurements` is already append-only in load order, so this is "first-loaded of the remainder" (REQ-PLOT-321) with no extra bookkeeping. This reassignment must run **between** `close_measurement`'s existing list-mutation line and its existing `_refresh_measurement_axes()`/`refresh_display_names()` tail calls, not appended after them — otherwise axes get pushed once against a stale/removed Primary reference with no second refresh to correct it. `set_primary_measurement(measurement)` (new, user-driven path) validates membership, updates the reference, and calls `_refresh_measurement_axes()` the same way.
- **REQ-PLOT-319 (Primary's axis row topmost) and the existing REQ-PLOT-312 (Sync reference) both fall out of one change: `_refresh_measurement_axes()` reorders its local list, Primary first, before pushing to `PlotStripesArea.refresh_measurement_axes()`.** Neither `PlotStripe.set_measurement_axes` (draws rows in whatever order it's given) nor `PlotStripesArea._update_x_axis_tick_visibility` (already reads `self._measurements[0]` as the sync reference — that's just "whatever `AppController` put first," an existing coincidence from #101/#102, not code that assumed load-order specifically) need to change at all. Reordering happens once, upstream.
- **`AppController` gains one `_refresh_signal_browser()` helper, called from every mutation site of `self._measurements`** (`replace_measurements`, `add_measurements`, `close_measurement`, `rename_measurement`) — `self._browser.populate_all([(m.label, m.loader.channel_tree()) for m in self._measurements])`. The browser receives `(label, groups)` pairs rather than `LoadedMeasurement` objects directly, mirroring `populate(groups)`'s existing plain-data contract — `SignalBrowser` has no reason to hold a `loader`-carrying object, only what it needs to render. Centralizing this was a deliberate fix to a gap the Plan-review pass caught: `add_measurements()` previously only called `self._browser.set_measurements()` (updates the combo), never `populate()` — harmless under the old switcher (a newly added measurement was only ever shown once the user switched the combo to it, lazily re-populating then), but silently wrong under the new always-fully-populated flat model, where a newly added measurement's channels must appear immediately without requiring any user action. One shared call site removes the class of "forgot to repopulate on this particular mutation path" bug entirely, rather than fixing this one instance and leaving the next mutation site free to reintroduce it.
- **Signal Browser's tree model is replaced by a flat, always-fully-populated model — `populate_all(measurements)` replaces `populate(groups)`/`set_measurements(labels)`.** Every channel from every loaded measurement becomes one row (REQ-BROWSER-010/011); `_LOCATION_ROLE` grows from `(group_index, channel_index)` to `(measurement_index, group_index, channel_index)`. One new role, `_SORT_ROLE`, holds the bare channel name (REQ-BROWSER-051 — sorting must ignore the `[M1]` display prefix); the channel's group name (REQ-BROWSER-013 — the channel-group hierarchy is removed as an organizing concept, but the information itself isn't discarded, just relocated to a hover tooltip) uses Qt's native `QStandardItem.setToolTip()` rather than a second custom role, since that's exactly what `Qt.ItemDataRole.ToolTipRole` already is — no reason to reinvent it. `_apply_filter`'s existing wildcard/substring logic is untouched (it already matches the full `DisplayRole` text, prefix included — no behavior change there); a new proxy subclass overrides `filterAcceptsRow` to additionally AND in the measurement-filter criterion (REQ-BROWSER-052/053), and overrides `lessThan` to compare `_SORT_ROLE` instead of `DisplayRole`.
- **The measurement combo is repurposed from switcher to filter — `_measurement_combo` → `_measurement_filter_combo`, gains an "All" entry at index 0.** `measurement_selected` (the signal) and `MainWindow._on_measurement_selected` are deleted entirely, not reused: today's switcher required a controller round-trip (re-`populate()` a different measurement's tree on selection); the filter is a pure view-local proxy narrowing with no controller involvement, the same as the existing text filter.
- **Drag/drop MIME payload flattens to a list of `(measurement_index, group_index, channel_index)` triples — `encode_signal_payload(items)` drops its separate top-level `measurement_index` parameter.** Required, not just a simplification, once a single multi-selection can legally span rows from different measurements (confirmed with the user: mixed-measurement drags are allowed in one gesture, matching double-click/"Add Signal" which already resolve per-row naturally). Verified there are exactly two consumers of `SIGNAL_MIME_TYPE`/`decode_signal_payload` (`ActiveSignalsTable`, `PlotStripe` — `ROW_MIME_TYPE` is a separate, unaffected mechanism), both needing the same change: `SignalBrowser.add_signals_requested`'s payload shape, `_DragTreeView` (loses its `get_measurement_index` callback entirely — each selected row already carries its own index via `_LOCATION_ROLE`), `signals_dropped_on_stripe` (emits the triple list instead of `(locs, stripe, measurement_index)`), and `MainWindow._on_add_signals`/`_on_add_signals_to_stripe` (resolve `measurement_at(mi)` per item inside the loop, not once before it).
- **MeasurementInfoBox becomes a `QTabWidget` wrapper, always tabbed (REQ-PLOT-318) — one tab per `LoadedMeasurement`, existing form content unchanged per tab, new header row above the form** holding the Primary checkbox and a short-name `QLineEdit`. Mutual exclusivity for "only one Primary" (REQ-PLOT-317) uses a `QButtonGroup(exclusive=True)` spanning every tab's checkbox — Qt6 exclusivity works for `QCheckBox` the same as `QRadioButton`, so user-driven "checking one unchecks the others" is native, not manually bookkept.
- **Implemented as a single `set_measurements(measurements, primary)` that always fully tears down and rebuilds every tab, rather than the incrementally-synced design first sketched here** (a targeted `set_primary(measurement)` that called `setChecked()` with signal-blocking to avoid a callback loop). The full-rebuild approach turned out simpler and just as correct: `AppController` gains a `_refresh_info_box()` helper mirroring `_refresh_signal_browser()`, called from every mutation site of `self._measurements` *and* `self._primary_measurement` (`replace_measurements`, `add_measurements`, `close_measurement`, `set_primary_measurement`, `rename_measurement`) — including a bare Primary change with no measurement added or removed. Since every call renders whatever `self._measurements`/`self._primary_measurement` currently are from scratch, the "controller-driven reassignment doesn't update the checkbox" gap this section originally called out (`QButtonGroup` auto-removes a *destroyed* button but doesn't auto-check a replacement) never arises — there's no stale state to leave behind, because nothing survives a rebuild to become stale. `rename_measurement()`'s duplicate-rejection path also just calls the same `_refresh_info_box()` on the `False` branch, which is what reverts the edit box's rejected text back to the still-current label — no separate revert mechanism needed. Current tab selection is preserved by index across a rebuild (captured before teardown, restored after, bounds-checked).
- **Tab teardown on every rebuild needs the same explicit `deleteLater()` discipline as #120's fix, applied proactively to a new site.** `QTabWidget.removeTab()` detaches but does not destroy the page widget, the same gap #120 found in tab-close (`MainWindow._on_tab_close_requested`) — `_teardown_pages()` removes each page's checkbox from the button group, `removeTab()`s the page, then `deleteLater()`s it, in that order (detaching from the button group first avoids a moment where an about-to-be-destroyed checkbox is still counted toward exclusivity). Lower blast radius than #120's native segfault (a leaked plain `QWidget` won't crash Qt the way an orphaned-but-connected `ViewBox` does — confirmed `QButtonGroup` cleanly auto-removes a deleted checkbox from itself, so membership doesn't block that checkbox's own `deleteLater()`), but the discipline is identical and cheap to apply consistently rather than reasoning "probably fine, it's not PyQtGraph." Because every rebuild (not just measurement-close) runs through this one teardown path, there's a single place to get this right instead of one per mutation site.
- **`close_measurement`'s existing per-signal and axis-row teardown paths are unaffected and already safe.** It already routes every affected `ActiveSignal` through `remove_signals()` → `PlotStripe.remove_signal()` (the `_destroy_vb_and_axis` path fixed in #120) and rebuilds axis rows through `_refresh_measurement_axes()` → `set_measurement_axes()` (also fixed in #120) — both of #103's pre-existing teardown-risk surfaces reuse code already hardened by #120's fix. The only genuinely new teardown surface #103 introduces is the MeasurementInfoBox tab, addressed above.
- **New File ▸ Close Measurement is a nested `QMenu` (submenu) on the File menu, rebuilt every time the parent menu's `aboutToShow` fires** — a new `_rebuild_close_measurement_menu()`, structurally mirroring but kept separate from the existing `_rebuild_recent_files()` (which inserts flat actions, not a submenu), one `QAction` per loaded measurement labeled with its short name. Selecting one runs the same has-active-signals confirmation dialog already used for tab/stripe close (REQ-FILE-028) before calling `AppController.close_measurement()`.
- **Primary/label state is explicitly not carried through `.mvc` persistence or Replace's active-signal-carryover flow (REQ-PLOT-322)** — resets to first-loaded on every `replace_measurements()`, the same as `offset_s` and Synchronized state already reset there (#101/#102 precedent). This is a deliberate scope decision surfaced by the Plan-review pass, not an oversight: REQ-FILE-030/031's "carry active signals over by name" preference governs per-*signal* cosmetic continuity across a same-file reload, a different concern from which *measurement* is Primary — Replace's new file set may have no relationship at all to the old one, so there is nothing meaningful to preserve identity-wise the way a same-file signal re-match has. Full multi-measurement `.mvc` persistence remains #106's job, matching #101's own deferral.

### Workspace/Configuration Format Extension (#106)

Extends `.mvc` session persistence to everything that shipped since it was last touched: per-tab Plot Stripes (#97), Main Widget Tabs (#99), and Multiple Measurements/Primary (#101/#102/#103) — today's format only ever captures the single active tab's flat signal list and the first measurement's path; every one of those features' own architecture entries explicitly deferred their persistence here.

- **`ViewerConfig` reshapes, not just grows.** Today it's flat and single-tab-implicit: `measurement_path: str` (singular), `signals: tuple[SignalConfig,...]` (no stripe/measurement reference), and `x_range`/`y_ranges`/`merged_groups`/`synced_groups`/`cursor_mode`/`cursor_positions`/`selected_signal` all live at the top level. New nested dataclasses: `StripeConfig` (name, size), `TabConfig` (name, `stripes: tuple[StripeConfig,...]`, active_stripe_index, and — moved down from `ViewerConfig`'s top level — `signals`/`x_range`/`y_ranges`/`merged_groups`/`synced_groups`/`cursor_mode`/`cursor_positions`/`selected_signal`, since all of those are properly per-tab state once tabs exist), `MeasurementConfig` (path, label, offset_s). `SignalConfig` gains `stripe_index: int = 0` and `measurement_index: int = 0`. `ViewerConfig`'s top level becomes: `format_version`, `measurements: tuple[MeasurementConfig,...]` (replacing the singular path), `primary_measurement_index`, `measurements_synchronized`, `tabs: tuple[TabConfig,...]`, `active_tab_index`, plus the unchanged display-name-rule/`window_geometry`/`splitter_sizes` fields. This is a real shape change, not an additive one — every prior format addition (#98's drawer keys, etc.) was "new optional field, defaults when absent"; moving `x_range` etc. from top-level into a list changes where a reader looks for them entirely.
- **`ConfigManager` needs an actual migration path on load, not per-field defaulting.** `save()` always writes the new nested shape (`"tabs": [...]`, `"measurements": [...]`), even for a 1-tab/1-measurement session — there is no reason to keep writing two shapes. `load()` detects an old flat file (no `"tabs"`/`"measurements"` keys, only the old top-level `"measurement_path"`/`"zoom"`/`"cursors"`/`"axes"`/`"selection"` keys) and synthesizes a single-element `TabConfig`/`MeasurementConfig` from them — a one-time structural translation, not a per-field `.get(key, default)` — so a session saved before #106 still loads into one tab, one stripe, one measurement, matching REQ-FILE-096/067's forward-compat precedent.
- **Restore runs in two phases, not one pass per tab — a Plan-review catch.** The first draft looped per-`TabConfig` (create tab → create stripes → resolve+add its signals → restore its axis grouping → next tab), which would batch near-match confirmation *per tab*, regressing the one-combined-dialog behavior REQ-FILE-095 requires (and that file-reload's existing `_restore_snapshots` already gets right for the multi-tab case). Correct order:
  - **Phase 0 — reset to one clean tab.** `remove_all()`/`replace_measurements()` only ever clear the *active* tab (`current.active.clear()`), not every tab — loading a saved multi-tab session over an already-multi-tab one would otherwise leave non-active tabs' signals pointing at just-discarded `LoadedMeasurement` objects. Tear down every tab but one before restore begins, the same as a fresh single-tab Replace already does for the one tab it touches.
  - **Phase 1 — load every saved measurement, index-aligned.** New `AppController` method (not a reuse of `replace_measurements()`, whose `LoadResult.succeeded` list drops failed paths and loses positional alignment with the saved measurement list — Plan-review catch): iterate the saved `MeasurementConfig`s, opening each into either a real `LoadedMeasurement` (label/offset_s assigned directly from the saved config, bypassing both `make_label()` and the single-interactive-rename `rename_measurement()`) or `None` at that same index on failure. If any are `None`, show the one combined "these couldn't be found" dialog (REQ-FILE-097/098) before continuing; canceling aborts the whole load. `_primary_measurement` is set from `primary_measurement_index` through this index-aligned list — if that slot is `None`, Primary falls back to the first-loaded of whatever remains, mirroring `close_measurement`'s existing reassignment (REQ-PLOT-321).
  - **Phase 2 — build every tab's skeleton (tab + stripes), no signals yet.** The tab that already exists at startup becomes the first restored tab (renamed, not recreated); each further `TabConfig` drives the same tab-creation factory `_on_new_tab()`'s "+" button uses. **Stripe double-creation is a second Plan-review catch**: `PlotStripesArea.__init__` already unconditionally creates one stripe, so naively calling `create_stripe()` `len(stripes)` times per tab leaves one extra empty un-renamed stripe — reuse that pre-existing first stripe (rename it to `stripes[0].name`) and call `create_stripe()` only `len(stripes) - 1` more times. Sizes are set once, only after every stripe for that tab exists (`create_stripe()` itself calls `self._splitter.setSizes([1]*len(self._stripes))` on *every* invocation — so setting sizes incrementally would just be overwritten by the next stripe's creation; the Plan-review pass confirmed this ordering was already correct in the draft, not merely cautious).
  - **Phase 3 — one combined classify/resolve/dialog pass across every tab's saved signals.** `_resolve_config_signals` (today single-tab, `measurement_aware=False`) and `_restore_snapshots` (today already multi-tab, `measurement_aware=True`, used by file-reload's "keep signals" flow) are structurally near-identical — same classify → exact/ambiguous/near-match branching → one shared `NearMatchDialog`/`SignalsNotFoundDialog` pass. Extract the shared classify+batch+dialog portion into one `_resolve_and_confirm_snapshots(snapshots_by_tab, measurement_aware) -> resolved_by_tab` helper used by both callers, rather than one calling the other outright — each caller still owns its own final "add to controller" step differently (`_restore_snapshots` calls `restore_tab_signals` directly; config-restore additionally needs to route each resolved signal into its saved `stripe_index` as part of adding it, via the equivalent of `move_signals_to_stripe`). `ActiveSignalSnapshot` gains `stripe_name: str = ""` (empty default = today's no-stripe-info behavior) to carry stripe assignment through the pipeline. Because Phase 1 already settled which measurements exist before Phase 3 runs, there's no rollback-semantics conflict between "a measurement failed to open" and "a signal failed to resolve" — Phase 3 runs only once every measurement reference is already known-valid, exactly matching `_restore_snapshots`'s existing assumption today (it already only ever runs after a file is known-loaded).
  - **Phase 4 — per tab: restore axis grouping, then zoom, then cursor, then selection** (this part is unchanged from today's single-tab order) **, then set that tab's active stripe; finally set the window's active tab.** Axis grouping must run after Phase 3 has routed every signal into its saved stripe, not before — `restore_axis_grouping()` → `merge_signals`/`sync_signals` → `_same_stripe_or_none()` silently no-ops a group whose signals aren't already in the same stripe (confirmed in code, and the exact reason today's single-tab restore code comments cite #106 by name).
- **Driving `_on_new_tab()`'s factory path N−1 times for bulk restore is a deliberate simplification, not an oversight.** It fires `switch_tab()` (drawer refresh, tab-bar repaint) on every call and doesn't reconcile `_tab_counter` against restored tab names — for a rare, one-time bulk operation (loading a saved session), the transient flicker and a cosmetic future "+"-created tab name collision (duplicate tab names are already possible today via manual double-click rename, so this isn't a new bug class) are an accepted tradeoff against building a separate non-interactive bulk-tab-creation path solely for this.
- **Stripe routing moved into `AppController.restore_signals()` itself, not a separate MainWindow-side step (M6).** The plan's Phase 3 sketch had `MainWindow` looking up each resolved signal's target `PlotStripe` object by name and passing it down to a stripe-aware add path. In practice `restore_signals()` already has direct access to `current.plot.get_stripes()`, so it does the by-name lookup itself from each `ActiveSignalSnapshot.stripe_name` and passes the result straight into `add_signal(stripe=...)` — a stripe object never needs to cross the controller/view boundary. This made the planned `MainWindow._find_stripe_by_name()` helper unnecessary (removed, along with its 3 dedicated tests) and, since `_snapshot_signals()` has captured `stripe_name` on every snapshot since M2 (including the pre-existing file-reload "keep signals" flow, `_restore_snapshots`), it activated stripe-preserving restore for that flow too — a signal reloaded via "keep signals" now goes back into the same-named stripe it was in before, not just whatever stripe happens to be active. This is a strict improvement (the field was captured but never consumed before M6), verified via full regression with no assertion changes needed in that flow's existing tests.
- **REQ-FILE-093 needed measurement-*scoped* search, not pool-wide search plus a disambiguation dialog.** The naive read of "resolves against that same measurement rather than an arbitrary one" could be satisfied by reusing the existing pool-wide `measurement_aware=True` search and letting the ambiguous-match picker sort it out when a name collides across measurements — but that would still prompt the user for something the saved config already unambiguously knows. Instead `_classify_signal_name()` gained a `measurement` parameter that, when given, searches only that one `LoadedMeasurement`'s own loader (`measurement.loader.find_signal_by_name`/`find_similar_signal_by_name`) and tags every candidate with that fixed measurement — matching the `(LoadedMeasurement, SignalMetadata)` shape `measurement_aware=True` already used, so `_resolve_and_confirm_snapshots()` didn't need a third candidate shape, just a per-item `effective_aware = measurement_aware or snapshot.measurement is not None` computed inside its loop (a snapshot can now individually opt into scoped search regardless of the caller's own `measurement_aware` flag). `ActiveSignalSnapshot` gained a `measurement: LoadedMeasurement | None = None` field for this — `None` for the reload flow (which has no prior knowledge of which measurement a re-added signal belongs to), set by config-restore from each `SignalConfig.measurement_index` resolved against Phase 1's `restore_measurements()` result. A signal whose `measurement_index` points at a `None` slot (that measurement failed to load) is folded straight into the not-found summary without attempting resolution at all — REQ-FILE-098's "folds any of its signals into the usual not-found summary."
- **Missing-measurement UX branches on count, inside `MainWindow._load_config()`, not inside the shared resolve helpers.** REQ-FILE-097 explicitly carves out an exception: "REQ-FILE-065 continues to apply unchanged when a session has exactly one measurement" — a zero- or one-measurement session still gets the old interactive `QFileDialog` locate-prompt on a miss, while a 2+-measurement session gets the new combined "these couldn't be found, continue or cancel" dialog with no per-file interactive prompt. `_resolve_saved_measurements()`/`_confirm_missing_measurements()` (built in M3) only implement the *combined* dialog path; `_load_config()` itself branches on `len(measurement_configs) <= 1` before choosing which path to take, keeping the count-based special case at the one call site that needs to know about it.

**Live-testing after the above landed surfaced three more real bugs and one genuine scope gap** — all root-caused and fixed within M6, before the milestone was considered done:

- **`TabConfig.y_ranges`/`merged_groups`/`synced_groups`/`selected_signal` all had the same bare-name ambiguity bug REQ-FILE-093 was written to prevent, just for the *other* references to a signal a saved tab makes** — the requirement's original wording only explicitly covered "which channel this resolves to" (Phase 3, the classify/near-match pipeline), but `_capture_tab()`'s zoom/grouping/selection capture used bare `active.metadata.name` as the sole key throughout, silently colliding whenever the same channel name was active from two different measurements in one tab. Confirmed by the user's own live-test report: a saved Merged group re-formed on live-testing "on a different signal group (same signal names but different source measurement)" — an exact description of a `dict`-key collision. Fixed by introducing `SignalRef(name, measurement_index)` — a saved reference to one specific signal — used in place of a bare `str` everywhere `TabConfig` points back at an active signal (`y_ranges` keys, each `merged_groups`/`synced_groups` member, `selected_signal`). `PlotStripe.get_axis_grouping()`/`PlotStripesArea.get_axis_grouping()` now return `(name, measurement)` pairs instead of bare names; `restore_axis_grouping()` matches on that pair. `AppController.restore_config()` gained a `measurements` parameter (Phase 1's index-aligned `restore_measurements()` result) to resolve each `SignalRef.measurement_index` back to the real `LoadedMeasurement` before matching.
- **That resolution must use `id()`, not the `LoadedMeasurement` object itself, as a dict key or equality operand.** `LoadedMeasurement` is a plain, non-frozen `@dataclass` — Python auto-disables `__hash__` on that combination (mutable + auto `__eq__`), so putting one directly into a dict key or set raises `TypeError: unhashable type` the moment more than one real measurement is loaded (a bare `MagicMock` stand-in in a test wouldn't have caught this — `MagicMock` is hashable by default — hence a dedicated regression test using the real loader-pool machinery). Separately, even where hashing wasn't needed (the selection match), `==` was wrong too, for the same reason `close_measurement`'s own precedent already established: `LoadedMeasurement`'s auto-generated field-equality `__eq__` would spuriously match two *distinct* measurements sharing the same path/label/offset. Both `AppController.restore_config()` and `PlotStripesArea.restore_axis_grouping()` now key exclusively by `id(measurement)` (or compare via `is`), never `==`/direct hashing.
- **Zoom desync between stripes after a session reload was actually the same root cause manifesting a third way** — not a separate pyqtgraph autoRange timing bug as first suspected (a plausible-looking but wrong theory, since `setXRange`/`setYRange` disabling autoRange is the same established pattern every other zoom operation in this codebase already relies on without special handling). Once `y_ranges`' bare-name collision was fixed, the "stripe zoom no longer matches" symptom resolved along with it.
- **Per-tab AST sidebar width (the plot|AST `QSplitter`) and the AST's own column widths were never captured at all** — a genuine scope gap, not a regression: REQ-FILE-090's original text only ever mentioned stripe layout, never the AST's own internal layout. Confirmed via code reading, not guesswork: `MainWindow._make_tab_page()` builds a *fresh* `QSplitter` per tab with a hardcoded `setSizes([500, _PANEL_W])` on every restore, and `ActiveSignalsTable`'s column widths are driven by one shared header (`_on_header_column_resized` already broadcasts any header resize to every stripe segment) with no capture/restore path at all. User explicitly chose to fold this into #106 now rather than file a follow-up (contrast with #105/#122/#119, filed separately during earlier milestones) — folded in as `TabConfig.page_splitter_sizes`/`ast_column_widths`, captured via new `MainWindow._tab_page_splitter_sizes()` (mirrors `_tab_names()`, since `AppController` has no access to the QSplitter either) and `ActiveSignalsTable.column_widths()`/`set_column_widths()`, restored in `_build_tab_skeletons()` and `AppController.restore_config()` respectively.

### Duplicate Tab / Copy Signals to new Tab (#119)

Two new tab-context-menu actions (REQ-PLOT-262–269): "Duplicate Tab" (full copy — signals, stripes, cursor, zoom, axis grouping) and "Copy Signals to new Tab" (signals only, flattened into one fresh stripe, no layout/view state).

- **Deliberately does NOT reuse #106's `TabConfig`/`SignalConfig`/`restore_config` pipeline.** That machinery exists to solve a different problem — reconciling a saved session against a *freshly reloaded* file, where signal identity is unknown and must be re-resolved by name (`SignalRef(name, measurement_index)`, near-match dialogs, `restore_measurements()`'s `loader.open()` I/O). A same-session duplicate already holds the exact `ActiveSignal`, `SignalData`, and `LoadedMeasurement` objects — routing through JSON-shaped snapshots would mean manufacturing fake `SignalConfig`s, re-running `loader.load_signal()` for data already in memory, and relying on name-matching that could in principle resolve to the wrong channel. Confirmed via a dedicated code survey before committing to this: `_capture_tab()`/`restore_config()` are entirely name/index-keyed by design, with no "already-resolved object" fast path anywhere in that pipeline. Instead, this feature clones `ActiveSignal` objects directly and drives the same lower-level view methods `AppController.add_signal()` itself calls (`PlotStripesArea.add_signal()`, `ActiveSignalsTable.add_row()`), skipping the loader entirely.
- **New tab positioning reuses the existing drag-reorder resync, not a new "insert at index" API.** The requirement (REQ-PLOT-264) is that the new tab lands immediately after the source tab, but `AppController.create_tab()`/`MainWindow._on_new_tab()` only ever append at the end (before the pinned "+" tab). Rather than adding an `insert_at` parameter to `create_tab()` and a parallel controller-side insertion path, the new tab is created normally (at the end) and then moved into place with one `self._tab_widget.tabBar().moveTab(new_index, source_index + 1)` call — `moveTab()` emits `tabMoved`, already wired to `_on_tab_bar_tab_moved()` (built for #99's manual drag-reorder), which re-pins the "+" tab and calls `AppController.reorder_tabs()` to resync `_workspaces` order. Zero new reordering logic; the existing, already-tested mechanism just gets triggered programmatically instead of by a mouse drag.
- **Stripe skeleton reuses `MainWindow._build_stripe_skeleton()` exactly as #106 built it**, fed freshly constructed (never-persisted, never touching JSON) `StripeConfig(name=stripe.name, size=size)` pairs built from the source tab's *live* `PlotStripe` objects (`plot.get_stripes()`/`get_stripe_sizes()`) rather than a `TabConfig` loaded from disk. Same "don't double-create the default stripe" discipline #106's Plan-review caught applies again: `PlotStripesArea.__init__` already creates one stripe, reused as `stripes[0]` rather than adding an extra one. "Copy Signals to new Tab" skips this step entirely — the destination's single auto-created default stripe *is* its one plot view, matching REQ-PLOT-267 (stripe layout explicitly discarded).
- **Signal cloning is one shared private helper, `AppController._clone_active_signal(old)`**, used by both new public methods below. Built via `dataclasses.replace(old, curve=None, view_box=None)` rather than listing every field by hand — `data`/`metadata`/`measurement` are shared by reference (no copy, no I/O; the new tab and the source share the same `LoadedMeasurement`, per REQ-PLOT-265's "share only the underlying measurement"), display attributes (`color`, `line_style`, `line_width`, `marker_shape`, `display_mode`, `step_mode`, the three `enum_display_*` flags) copy by value automatically, and `curve`/`view_box` are explicitly reset to `None` so `PlotStripesArea.add_signal()` builds fresh ones for the new plot — mirroring exactly how a normal `add_signal()` call constructs an `ActiveSignal` today. `dataclasses.replace` also means a future new `ActiveSignal` field is cloned automatically with no edit needed here, unlike a hand-written field list.
- **Zoom-state restore needs an object-identity remap; axis-grouping restore does not — an asymmetry worth calling out explicitly**, since it's easy to assume both work the same way. `PlotStripe.ZoomState.y_ranges` is a `dict[ActiveSignal, tuple]` keyed by the *exact object* whose `view_box` it read (`get_zoom_state()`/`set_zoom_state()` in `plot_stripe.py`) — calling `dest.plot.set_zoom_state(source_state, dest.active)` directly would silently restore nothing, since none of `dest.active`'s (new, cloned) objects are keys in a dict built from the *source's* objects. The clone loop therefore builds `old_to_new: dict[ActiveSignal, ActiveSignal]` alongside cloning, and zoom restore remaps `y_ranges` through it before calling `set_zoom_state()`. `PlotStripesArea.get_axis_grouping()`/`restore_axis_grouping()`, by contrast, already match by `(name, id(measurement))` (built for #106 M6's cross-measurement bare-name-collision fix) — since a clone keeps the same `metadata.name` string and the same `measurement` object reference, the source's captured `(name, measurement)` pairs already resolve correctly against the destination's cloned signals with no translation needed. Cursor restore (`CursorController.snapshot()`/`restore()`) needs no remap either — it's just `{mode, positions}`, no per-signal reference at all.
- **New `AppController` methods, `duplicate_tab_signals(source_index, dest_index)` and `copy_signals_to_new_tab(source_index, dest_index)`**, both assuming the destination workspace/tab already exists (view-side has already created the tab, and — for Duplicate — already built its stripe skeleton) — mirroring `restore_config()`'s existing assumption that skeleton-building is a separate, earlier step it doesn't itself perform. `duplicate_tab_signals()` walks `zip(source.plot.get_stripes(), dest.plot.get_stripes())` (already matched 1:1 by the skeleton step) and clones each stripe's signals across, then restores zoom/grouping/cursor. `copy_signals_to_new_tab()` flattens every source stripe's signals in `get_stripes()` order (top-to-bottom), into the destination's single stripe (REQ-PLOT-268) — no zoom/grouping/cursor restore at all (REQ-PLOT-267). Both end by setting `dest.color_index = source.color_index` (REQ-PLOT-269); neither touches `dest.selected`/`selected_signals` (REQ-PLOT-259/266's "no selection carried over" — already true of a freshly built `TabWorkspace`, nothing to reset) or `dest.zoom_ctrl` (a fresh `ZoomController` instance already has empty undo history; `set_zoom_state()` writes ranges directly to the plot, bypassing undo tracking entirely, so "fresh undo history" falls out for free rather than needing an explicit reset).
- **Per-stripe signal order must come from `workspace.active`, not `PlotStripesArea.get_signals_in_stripe()` — a real bug caught by architecture review before implementation.** `get_signals_in_stripe()` returns signals in `_signal_stripe` dict *insertion* order, which a same-stripe row drag (`ActiveSignalsTable._apply_row_move` → `AppController.reorder_signals()`) never updates — only `workspace.active`'s own list order changes on an in-stripe reorder. `_capture_tab()` (#106) already gets this right by sourcing order from `workspace.active` filtered to one stripe via `plot.get_stripe_for_signal()`; both new methods here do the same (`[a for a in source.active if source.plot.get_stripe_for_signal(a) is stripe]`) rather than reusing `get_signals_in_stripe()`, or a prior drag-reorder would silently be lost on duplicate/copy.
- **Both new methods end with a Z-order refresh on the destination workspace — another review catch.** `AppController.add_signal()` always calls `self.refresh_z_order()` afterward; Z-values are otherwise only ever set inside `PlotStripe.set_selected_signals()`, never inside `add_signal()` itself. Since both new methods drive `PlotStripesArea.add_signal()`/`ActiveSignalsTable.add_row()` directly (bypassing `AppController.add_signal()` entirely, by design — no loader I/O), they must each call the same Z-order refresh explicitly at the end, or cloned curves render at an arbitrary stacking order until something unrelated touches that tab.
- **`tabBar().moveTab(new_index, source_index + 1)` no-ops when the source is the currently-last real tab** (`source_index + 1 == new_index`, so Qt sees `moveTab(x, x)` and doesn't emit `tabMoved`). Harmless today only because `create_tab()` already appends to `_workspaces` in matching order, so the "already correctly positioned" case needs no resync — but this reliance is deliberately covered by an explicit regression test (duplicate/copy the last real tab) rather than left as an undocumented assumption.
- **Menu enablement and naming stay in `MainWindow`**, reusing existing primitives rather than adding new controller-side checks: "Copy Signals to new Tab" is disabled via the same `controller.tab_has_signals(index)` the close-tab-warning already uses (REQ-PLOT-263); the new tab's name is set with a plain `self._tab_widget.setTabText(new_index, "Copy of " + source_name)` — always prepended, no dedup against an already-"Copy of "-prefixed source name (REQ-PLOT-264), no new naming infrastructure beyond the existing `setTabText()` call `_rename_tab()` already uses.

### Replace a Single Measurement (#122)

Lets the user swap the file backing one already-loaded measurement (e.g. reload a corrected "M2") without disturbing any other loaded measurement (REQ-FILE-100–108).

- **`LoadedMeasurement` is mutated in place — `.loader`/`.info` are swapped on the existing object, never removed-and-reinserted into `AppController._measurements`.** This is the load-bearing decision the rest of the feature falls out of: `.label`, `.offset_s`, membership in `_measurements_synchronized`'s scope, and `is self._primary_measurement` are all identity-based today (see `close_measurement`'s own `m is not measurement` comparisons and #103's "Primary is one reference, not a per-instance flag"). Keeping the same object means short name, load-order position, X-axis offset, Synchronized membership, and Primary status (REQ-FILE-103) all carry over with **zero** extra bookkeeping — there is no new "reassign Primary if it was this one" branch to write or get wrong, unlike `close_measurement`'s REQ-PLOT-321 reassignment, because the identity never changes in the first place.
- **New `AppController.replace_single_measurement(measurement, path) -> LoadResult`.** Opens *path* into a fresh `self._loader_factory()`-built loader **before** touching anything else — on `MdfLoadError`, returns immediately with the failure in `LoadResult.failed` and *measurement* completely untouched (REQ-FILE-106), mirroring `add_measurements()`'s "a failure never affects any already-loaded measurement" contract (REQ-FILE-024) rather than `replace_measurements()`'s "failure leaves nothing loaded" contract (REQ-FILE-041) — the two existing Replace/Add failure modes already cover both shapes this method needs to choose between. Only once the new loader opens successfully does it: tear down *measurement*'s own active signals across every tab via the exact `affected = [a for a in workspace.active if a.measurement is measurement]` / `remove_signals(affected)` loop `close_measurement()` already uses (reusing the #120-hardened teardown path verbatim, not a new removal mechanism); assign `measurement.loader = loader; measurement.info = loader.measurement_info()`; add the path to Recent Files; and run the same `_refresh_signal_browser()` / `_refresh_info_box()` / `_refresh_measurement_axes()` / `refresh_display_names()` tail calls every other pool-mutating method already ends with. Does **not** touch `color_index` (matches `add_measurements()`'s reasoning — other measurements' already-active signals must keep their existing colors) or any tab's `cursor_ctrl`/`zoom_ctrl` (unlike whole-pool `replace_measurements()`'s reset — other measurements' signals remain live in the same tabs throughout, so blanket-resetting cursor/zoom would discard state that was never part of this operation; the removed/re-added signals' own cursor refresh already happens inside `remove_signals()`/`restore_signals()`).
- **New `AppController.snapshot_measurement_signals(measurement) -> dict[int, list[ActiveSignalSnapshot]]`, scoped to one measurement's active signals across every tab.** `AppController._snapshot_signals(workspace)` gains an optional `actives: list[ActiveSignal] | None = None` parameter (defaults to `workspace.active`, preserving every existing caller unchanged); the new method filters each tab's `active` list to `a.measurement is measurement` first, then tags every resulting snapshot with `dataclasses.replace(snap, measurement=measurement)`. That `.measurement` field already exists on `ActiveSignalSnapshot` for exactly this purpose (#106 M6, REQ-FILE-093) — `MainWindow._classify_signal_name()` already special-cases a snapshot carrying its own `.measurement` to scope resolution to that one `LoadedMeasurement`'s channel tree, and `_resolve_and_confirm_snapshots()`/`_restore_snapshots()` already consume that shape unchanged. Result: single-measurement carry-over (REQ-FILE-104) reuses the entire by-name/near-match/ambiguous-picker/not-found pipeline (REQ-FILE-030–036) with **no new resolution code at all** — only a new, scoped snapshot producer.
- **`MainWindow._replace_single_measurement(measurement)` mirrors `_load_files()`'s existing Replace shape, scoped to one measurement.** `QFileDialog.getOpenFileName` (singular — REQ-FILE-102, not `getOpenFileNames`); a new `_collect_measurement_snapshots_if_keeping(measurement)` mirrors `_collect_snapshots_if_keeping()` exactly (same `keep_signals_on_load` preference branches: `"never"` short-circuits to `{}`, `"ask"` shows the same "Keep Active Signals?" `QMessageBox`, guarded first by `controller.measurement_has_signals(measurement)` instead of `tab_has_signals(index)` for every tab) but sources its dict from `snapshot_measurement_signals(measurement)` instead of `snapshot_tab_signals(index)`; calls `controller.replace_single_measurement(measurement, path)`; on `result.failed`, the same `QMessageBox.critical` one-liner `_load_files()` already uses; on success with a non-empty snapshot dict, `_restore_snapshots(snapshots)` **completely unchanged** — it already only depends on each snapshot's own `.measurement` field, never on how the dict was scoped.
- **Two entry points, both thin wrappers around `_replace_single_measurement`.** (1) File ▸ Replace Measurement: a new `_rebuild_replace_measurement_menu()` structurally identical to `_rebuild_close_measurement_menu()` (one `QAction` per `measurement.label`, rebuilt on the File menu's `aboutToShow`, same signal-collision-safe `lambda checked=False, m=measurement: ...` capture), added next to the existing Close Measurement submenu. (2) A "Replace…" button in `_MeasurementInfoPage` (measurement_info_box.py), next to the existing Primary checkbox and short-name field — `MeasurementInfoBox` gains a `replace_requested = pyqtSignal(object)` alongside its existing `primary_change_requested`/`rename_requested`, wired in `MainWindow.__init__` the same way those two already are. Deliberately **not** a context menu: there is no existing per-measurement row to right-click anywhere (Signal Browser rows are individual channels post-#103's flattening, and the Info Box's only per-measurement UI is its tabs) — a button next to the other per-measurement controls already on that tab fits the established pattern with no new interaction mechanism.
- **A "Close" button is added to `_MeasurementInfoPage` at the same time, for symmetry with "Replace…" (REQ-FILE-108), as an additional entry point only — no behavior change.** `MeasurementInfoBox` gains a matching `close_requested = pyqtSignal(object)`; `MainWindow`'s handler is the existing `_on_close_measurement_requested(measurement)` body (has-active-signals confirmation, then `controller.close_measurement()`) reused as-is, just called from a second signal connection — not duplicated.
- **No new PyQtGraph/Qt teardown surface.** The only object torn down is via `remove_signals()`, already routed through the #120-hardened `PlotStripe.remove_signal()` path; the file/loader swap itself only reassigns plain Python attributes (`LoadedMeasurement.loader`, `.info`) on an object nothing else deletes or reparents. Checked explicitly against the #120 postmortem's checklist before finalizing this design, precisely because that pitfall is easy to reintroduce silently in any code that removes-then-re-adds signals.
- **`docs/ui.md` needs the File ▸ Replace Measurement submenu and the Info Box's new "Replace…"/"Close" buttons documented** in the same commit as the implementation, per the project's UI-doc rule — not done as part of this architecture pass, which only covers `docs/architecture.md`/`docs/requirements/file-handling.md`.

### Apply Config to Already-Loaded Measurements (#105)

Lets a saved `.mvc` workspace (tabs, stripes, signal selections) be applied onto whichever measurement(s) are already loaded, instead of opening the files the config itself refers to (REQ-FILE-110–119).

- **Central finding, confirmed by re-reading #106's own restore pipeline before designing anything new: three of its four phases are already measurement-source-agnostic and need zero changes.** `MainWindow._reset_to_single_tab()` (Phase 0) only touches `AppController._workspaces`/the `QTabWidget` — its own `AppController.remove_all()`/`remove_tab()` calls never read or write `self._measurements`. `_build_tab_skeletons()` (Phase 2) only builds tabs/stripes from `TabConfig`s, no measurement I/O at all. `AppController.restore_config()` (Phase 4) already takes an explicit `measurements: list[LoadedMeasurement | None]` param, index-aligned to `config.measurements`, and never mutates `self._measurements`/Primary/Sync — it was already written to resolve `SignalRef`s against *whatever* list it's handed (originally so a failed-to-load measurement could still be represented as `None` at the right index, REQ-FILE-098). The **only** phase that assumes "the measurements list came from opening files" is Phase 1, `AppController.restore_measurements()` — and that's the one phase #105 replaces outright, with a pure view-side selection that never touches the model layer or performs any I/O.
- **New `view/measurement_mapping_dialog.py` (`MeasurementMappingDialog`)** — one `QComboBox` per `config.measurements` entry (labeled with that slot's saved short name and its recorded file's bare name, e.g. `Path(mc.path).name`, purely for the user's own identification since that file is never opened, REQ-FILE-112), each combo populated with a **static, never-rebuilt** item list — "None" plus every entry of `controller.measurements` (by current live `.label`), always all of them, in every row. Defaults each combo to the live measurement at the same load-order position when one exists there, else "None" (REQ-FILE-113). Enforces 1:1 uniqueness by **override, not exclusion**: on any combo's `currentIndexChanged`, if the newly-selected value is a real measurement (not "None"), every *other* combo currently holding that same measurement is reset to "None" (signals blocked during the reset). A first cut excluded already-chosen measurements from other rows' item lists instead — live-tested and rejected: it made swapping two already-assigned slots impossible without first freeing one via "None", since the desired target was never offered as a choice at all. The override model fixes this directly — every row always offers every measurement, and the most recent selection always wins — while still guaranteeing at most one slot maps to any given measurement. `OK`/`Cancel` via a plain `QDialogButtonBox`; `.mapping() -> list[LoadedMeasurement | None]` returns the index-aligned result Phase 2–4 already expect. Skipped entirely (proceeds with `mapped = []`) when `config.measurements` is empty — nothing to map.
- **New `MainWindow._on_apply_config()`** is the only new orchestration method, and it's a thin re-sequencing of pieces that already exist: `QFileDialog.getOpenFileName` filtered to `.mvc` only (new shared `_MVC_FILE_FILTER` module constant, also adopted by `_on_save_config_as()` to remove the duplicate inline filter string) → `ConfigManager.load(path)` (pure JSON parsing, confirmed to perform no I/O on the measurement paths it records — nothing to guard against here) → `MeasurementMappingDialog` (cancel aborts the whole action, REQ-FILE-114) → `_apply_window_geometry`/`_apply_splitter_sizes` (workspace-level, applied the same as a normal session load, REQ-FILE-115) → `_reset_to_single_tab()` → `_build_tab_skeletons(config.tabs)` → `_resolve_config_signals_for_tabs(config.tabs, mapped)` (unchanged — REQ-FILE-117/118 fall out of reusing this exactly, since a `None` mapping already produces the same not-found behavior REQ-FILE-098 established for a failed-to-load measurement) → `SignalsNotFoundDialog` if needed → `controller.restore_config(config, resolved_by_tab, mapped)` → sync `_tab_widget`'s current index to `config.active_tab_index` (mirrors `_load_config`'s own tail). Deliberately never assigns `controller.current_config_path = path` (REQ-FILE-119) — unlike `_load_config()`, which does exactly that for a normal session load — so a later plain "Save Workspace" can't silently overwrite the applied template with a different measurement mapping; instead calls `self._on_save_config_as()` unconditionally right after, whose own cancel path already does nothing (existing behavior, not new).
- **Confirms the REQ-FILE-116 "measurement pool untouched" requirement is true by construction, not by any new guard.** No new code reads or writes `AppController._measurements`, `_primary_measurement`, or `_measurements_synchronized` anywhere in this feature — the reused phases already didn't touch them (this was the whole point of re-auditing them first), and the new mapping dialog only ever *reads* `controller.measurements` to populate its combos.
- **Entry point:** a new `_apply_config_action` in the File menu, placed directly after "Open…" (grouping the two file-dialog-driven "bring content in" actions together, ahead of the Save Workspace pair). Enabled state is recomputed lazily on the File menu's `aboutToShow` — a new `_update_apply_config_enabled()` mirroring the existing `_rebuild_close_measurement_menu`/`_rebuild_replace_measurement_menu` lazy-refresh pattern, rather than threading an enabled-state update through every measurement-pool-mutating call site — `setEnabled(controller.measurement_count >= 1)`.
- **Deliberately not reusing `_resolve_saved_measurements()`/`_confirm_missing_measurements()`** (the "locate a missing file on disk" pair) — those solve "the config's own recorded path can't be found," a problem that doesn't exist here since no path is ever opened. The new mapping dialog is a different, additive concern: "which already-loaded measurement should each slot's signals resolve against."
- **`docs/ui.md`/`docs/api.md` need the new File ▸ Apply Config… entry, `MeasurementMappingDialog`, and `MainWindow._on_apply_config()` documented** in the same commit as the implementation, per the project's UI/API-doc rules.

### Signal Visibility (#133)

Lets an active signal be temporarily hidden (curve + its own Y-axis) without removing it from the Active Signals Table or affecting any other display setting (REQ-PLOT-330–339).

- **`ActiveSignal` gains `visible: bool = True`.** Same mutable-field pattern as every other per-signal display property (`line_width`, `step_mode`, ...) — set by the controller, applied to the plot by the view, never inferred from Qt widget state.
- **A pre-existing mechanism already does 80% of this feature: the "Show Only Selected Y-Axis" toggle's `PlotStripe._update_axis_visibility()`.** It already computes, per shared axis (merged-group-aware), whether *any* member of that axis's group should be visible, and does the actual layout work to reclaim column space — `self._pi.layout.removeItem(axis)` then `axis.hide()` (and the reverse: `layout.addItem(axis, 2, col)` then `axis.show()`). This is exactly REQ-PLOT-335/336's "hide the axis column too, unless another group member is still visible" — composing the two features only required changing `_update_axis_visibility()`'s `visible_set` computation to also subtract hidden signals (`{a for a in self._data if not a.visible}`) before applying the existing selected-only narrowing, rather than writing a second, parallel hide/show mechanism. Confirmed this composition is correct for **Merged** groups (one shared ViewBox+axis object across members, so "any member visible" is the right OR-condition) — **Synced** groups don't share axis objects at all (each member keeps its own independent ViewBox/axis, only Y-range is externally forced equal), so a Synced member's own visibility only ever affects its own axis, with no cross-member interaction needed; REQ-PLOT-336's "Merged or Synced" framing is still accurate at the requirements level since a Synced-only member is trivially a one-member group axis-wise.
- **`PlotStripe.set_signal_visible(active, visible)`** (new) — `spd.curve.setVisible(visible)`, then `self._update_axis_visibility()`. `PlotStripesArea.set_signal_visible(active, visible)` (new) dispatches to the owning stripe, mirroring `set_step_mode`'s existing dispatch-by-`_signal_stripe` pattern exactly.
- **The curve's own `ViewBox`/`AxisItem` objects are never destroyed or recreated by this feature — only shown/hidden.** This was a deliberate choice, checked against the #120 postmortem before finalizing: reusing `layout.removeItem()`+`.hide()`/`layout.addItem()`+`.show()` (already-working code from the Show-Only-Selected feature) means there is no new teardown surface at all, and the `ViewBox` stays alive and correctly positioned (via `_update_view_geometries()`, unaffected by axis visibility) the entire time a signal is hidden — which is exactly what makes cursor value interpolation and the floating enum cursor label keep working correctly for a hidden signal (REQ-PLOT-338) with **zero code changes** to `model/interpolate.py` or `cursors.py`: both operate on `active.data`/`view_box` coordinate mapping, neither of which cares whether the axis or curve widget is currently shown.
- **`PlotStripe._display_units()` gains one filter line** — `signals = [s for s in signals if s.visible]`, applied right after establishing the candidate signal list (whether the default `list(self._data)` or an explicitly-passed `ordered_signals`). Since `autorange_y()`, `zoom_y_to_view()`, and `swimlanes()` all already source their per-axis grouping from this one shared helper, this single change satisfies REQ-PLOT-337 for all three at once — a partially-hidden Merged group's unit is still formed, just from only its visible members' data, and a fully-hidden group forms no unit at all (nothing to autorange/arrange, which is correct since it isn't shown). `PlotStripesArea.zoom_to_fit()`'s own X-range computation doesn't route through `_display_units()` (it scans `self._signal_stripe` directly across the whole area) and needs the identical `.visible` filter added directly, with a guard returning early if zero signals remain visible (mirrors the existing "`if not signals: return`" empty-pool guard already there). `PlotStripe.zoom_to_fit()` (the per-stripe method, not the production `PlotStripesArea` entry point but kept consistent for direct callers/tests) gets the same filter+guard.
- **New Active Signals Table column, placed first — before the color swatch, leftmost.** AST column widths persist to `.mvc` purely by index (`ActiveSignalsTable.column_widths()`/`set_column_widths()` iterate `range(columnCount())`/`enumerate(widths)`, no name-keyed lookup), so inserting a column at the front does shift every existing column's saved width index by one on restore from an old file — flagged explicitly and accepted (user's call): the effect is purely cosmetic and self-correcting (columns render at a stale-but-harmless width until manually resized once), not a crash or data-loss risk, and prominent/scannable placement won out over avoiding that one-time resize. `_COL_VISIBLE = 0`, every other `_COL_*` constant shifts down by one (`_COL_COLOR = 1`, `_COL_NAME = 2`, etc.); `_NUM_COLS` 5→6; `_HEADERS` gains a new leading `""` (icon-only, like the color column's own empty header); `Fixed` resize mode sized like `_COL_COLOR`.
- **New `VisibilityToggleButton(QPushButton)` in `view/widgets/` (new `visibility_toggle_button.py`, exported from `view/widgets/__init__.py`)** — mirrors `ColorSwatch`'s exact shape (`QPushButton` subclass, `setFixedSize`, `setFlat(True)`), but swaps between the already-present `eye_open`/`eye_hidden` icon assets (light/dark variants, `@2x`, all already in `resources/icons/` — no new asset work needed) via a `set_visible_state(bool)` method, constructed per-row in `_add_row_to_segment` the same way `swatch = ColorSwatch(active.color)` already is.
- **`_load_icon()`/`_icon_suffix()` extracted from `main_window.py` into a new `view/widgets/icons.py`.** Required, not optional: `active_signals_table.py` needs them for the new button too, and it can't import them from `main_window.py` (which already imports `ActiveSignalsTable` — a reverse import would be circular). `main_window.py` switches to `from mdf_viewer.view.widgets.icons import _load_icon, _icon_suffix`; since a plain `from X import Y` still binds `Y` as an attribute of the importing module, the two existing tests that call `main_window._icon_suffix()` directly keep passing unchanged. `_ICONS_DIR`'s relative path adjusts from `parent.parent` to match the new file's one-deeper location (`view/widgets/` vs `view/`).
- **One new `ActiveSignalsTable` signal, `visibility_toggle_requested = pyqtSignal(list)`, with three emission sites converging on it** — the eye button's `clicked` handler, a new "Toggle Visibility" Active Signals Table context-menu entry, and a new `Ctrl+W` case in the existing `keyPressEvent` override (alongside the current `Delete`/`Backspace` → `_on_remove_clicked()` handling). All three resolve their target list via the **already-existing** `_selected_signals()` / "apply to the whole selection if the acted-on row is part of one" logic `_on_color_swatch_clicked()` already implements (`targets = selected if len(selected) > 1 and active in selected else [active]`) — confirms REQ-PLOT-333/334's button-vs-selection scoping is exactly this codebase's existing precedent for a per-row button that can also act on a multi-selection, not a new interaction model.
- **New `AppController.toggle_signal_visibility(actives: list[ActiveSignal])`** — for each active still in `current.active`: flips `active.visible`, calls `current.plot.set_signal_visible(active, active.visible)`, then `current.table.set_row_visible_icon(active, active.visible)` (new `ActiveSignalsTable` method mirroring `set_row_color`, updating the button's icon after a toggle that didn't originate from clicking that exact button — e.g. Ctrl+W or the context menu). Wired in `MainWindow._wire_tab(...)` alongside every other per-row action: `active_signals_table.visibility_toggle_requested.connect(controller.toggle_signal_visibility)`.
- **Persistence threads `visible` through the same four places every other per-signal display property already goes through, all as an optional field defaulting to `True` (REQ-FILE-067's forward-compat convention, so an old `.mvc` file with no `visible` key still loads every signal visible):** `ActiveSignalSnapshot.visible` (set from `active.visible` in `_snapshot_signals()`); `SignalConfig.visible` (`model/viewer_config.py`, set from `snap.visible` in `_capture_tab`'s `SignalConfig(...)` construction); `ConfigManager`'s JSON (de)serialization (`"visible": s.visible` on save, `visible=bool(s.get("visible", True))` on load); and `AppController.restore_signals()`, which gains one more `if snap.visible != active.visible:` branch alongside the existing line-width/line-style/display-mode/step-mode reconciliation block, calling `set_signal_visible` + updating `active.visible` + the AST icon the same way. `MainWindow._signal_config_to_snapshot()` (#105/#106) passes `visible=sig.visible` through unchanged, same as every other field it already forwards.
- **No new PyQtGraph/Qt teardown surface** — every object this feature touches (`curve`, `axis`, `ViewBox`) already existed and is only ever shown/hidden, via the same `layout.removeItem()`+`.hide()` / `layout.addItem()`+`.show()` pair the Show-Only-Selected-Y-Axis feature already exercises correctly today. Checked explicitly against the #120 postmortem's checklist before finalizing, per this project's standing practice for any change touching per-signal plot object lifecycle.
- **`docs/ui.md`/`docs/api.md` need the new AST column, `VisibilityToggleButton`, the context-menu entry, and the Ctrl+W shortcut documented** in the same commit as the implementation.

### Session Persistence Extraction (#136)

Found during a whole-codebase architecture review (2026-07-12): the `.mvc` save/load/apply orchestration (~600 lines) was split across `MainWindow` and `AppController`, each reaching into the other constantly, with `_load_config`/`_on_apply_config` as near-duplicate 5-phase pipelines and a shared resolution helper carrying a `measurement_aware` flag whose docstring admitted it existed only to "preserve each existing caller's exact prior behavior" post-merge.

- **One new class, view-layer only: `view/workspace_session_controller.py::WorkspaceSessionController`.** No new controller-side class — `AppController`'s own config methods (`capture_config`, `restore_measurements`, `restore_config`, ...) were already cohesive, take their view-owned inputs as plain parameters, and never reach back into `MainWindow`; the entanglement was asymmetric, concentrated in `MainWindow`'s orchestration. Despite the "-Controller" suffix (kept to match the issue's own wording), the class lives in `view/`, not `controller/`, because it constructs/shows concrete `QDialog`s and manipulates `QTabWidget` pages directly — both prohibited in `controller/` by this project's MVC rule (see "Layers" table above and "View imports in controller: TYPE_CHECKING-only").
- **Unified `_load_config`/`_on_apply_config` into one real shared `_restore_saved_workspace`**, parameterized only by an `obtain_measurements` strategy callable — file-open + `restore_measurements()` for `load_config()`, vs. a `MeasurementMappingDialog` selection for `apply_config()`. A Plan-review catch during design: the shared body must **not** call `reset_to_single_tab()` itself — each strategy closure calls it internally, at exactly the point the original code did, for two reasons confirmed against the actual pre-extraction code: (1) `restore_measurements()`'s own docstring assumes the controller is already down to a single clean tab, so it must run *after* the reset, and (2) both original methods return early on a cancelled dialog *before* ever resetting, so the user's open tabs survive a cancelled Load/Apply — hoisting the reset out of that guard would have destroyed the workspace on every cancel, a real behavioral regression rather than a cosmetic reordering.
- **Dropped `_resolve_and_confirm_snapshots`'s `measurement_aware` parameter** (confirmed dead: exactly two real call sites in the whole codebase, both hardcoded `measurement_aware=True`). Kept `use_group_name` — a genuine behavioral difference between session-restore (narrows by saved `SignalConfig.group_name`) and the unrelated reload-"keep signals" feature (never narrows by group), not accidental complexity from the flag's own docstring history.
- **`_resolve_and_confirm_snapshots`, `_classify_signal_name`, `AppController._snapshot_signals`/`restore_signals`/`restore_tab_signals` all stay where they were** — shared general infra with the unrelated file-reload "keep signals" feature, not session-persistence-only.
- **`MainWindow` keeps every previously-private method name as a one-line delegate to `self._session`.** Dominant risk-reduction choice: tests patch/call these by attribute name on the `MainWindow` instance, so preserving names meant ~100 existing tests needed no changes for the pure method-relocation steps.
- **`_reset_to_single_tab`'s `removeTab`/`deleteLater`/`remove_tab`/`remove_all` teardown sequence moved verbatim**, its own isolated commit, diffed character-for-character against the original before proceeding — the one step in this extraction touching the #120-class Qt-teardown-leak crash area. Full regression suite plus a live smoke test (multi-tab session with signals, load a different `.mvc` file over it) required before considering that step verified, per the #120/#78 live-verification rule.
- **A real bug found via live testing, not caught by any automated test beforehand:** the delegate-preserving design above does *not* by itself protect calls made *internally* by `WorkspaceSessionController` back out to a `MainWindow`-owned method — those were originally wired as bound methods captured once in `MainWindow.__init__` (e.g. `save_config_as=self._on_save_config_as`). A bound method captured at construction time cannot be redirected by a later `patch.object(window, "_method_name")` — that call only shadows the instance attribute, and Python's bound-method object already closed over the original unpatched function. This meant `WorkspaceSessionController.apply_config()`'s `finalize()` step called the *real* `_on_save_config_as()`, which hit the real, unmocked `QFileDialog.getSaveFileName()` — and under headless/offscreen Qt, a real dialog's `.exec()` blocks forever with nothing able to close it, since offscreen only suppresses rendering, not the modal event loop. Surfaced as a live pytest process hanging for 17 minutes with no visible window (the user noticed the elapsed time, not a dialog — offscreen dialogs are invisible by design), tracked down by bisecting the test batch down to a single hanging test ID and reasoning through the exact call chain rather than guessing. Fixed by wiring every `MainWindow`-owned dependency `WorkspaceSessionController` needs as a lambda that looks up the method on `self` (the `MainWindow` instance) at *call* time, not construction time — matching the pattern already used for `get_controller`/`get_settings` (which read `self._controller`/`self._settings` live because those start `None` and are wired post-construction). One existing test (`test_apply_config_success_calls_restore_pipeline`) asserted directly on `wired._reset_to_single_tab`/`wired._build_tab_skeletons` being called during `apply_config()` — those calls happen on `wired._session` directly (a genuine same-class internal call, not routed through a `MainWindow` lambda), so its patch targets moved to `wired._session.reset_to_single_tab`/`.build_tab_skeletons`.
- The extraction itself (steps above) is a pure structural refactor with no behavior change, confirmed by the full 1719-test suite passing unchanged before and after — no `docs/ui.md` change and no `CHANGELOG.md` entry for that part, per `CLAUDE.md`'s Issue Triage rule for Tech debt/Refactor issues.
- **A separate, pre-existing Bug caught via live testing of the extraction, fixed in the same session (not part of #136's own scope, but adjacent and small):** cancelling Apply Config's `MeasurementMappingDialog` still visibly resized the window, violating REQ-FILE-114's "cancelling this dialog aborts the entire Apply Config action with nothing changed." Confirmed via `git show` against the pre-#136 code that `_apply_window_geometry`/`_apply_splitter_sizes` ran unconditionally before the mapping dialog in the *original* `_on_apply_config` too — not a #136 regression, just faithfully preserved by it. The existing `REQ-FILE-114` test (`test_apply_config_mapping_cancel_aborts`) only asserted `restore_config` wasn't called, never checking geometry, so the gap had no coverage. Fixed by moving geometry/splitter-size application out of the shared `_restore_saved_workspace` entirely, back into each caller at its own correct point: `load_config()` still applies them unconditionally up front (no equivalent requirement for that path, and an existing test — `test_load_config_applies_saved_window_geometry` — already asserts that ordering deliberately); `apply_config()`'s `obtain_measurements` closure applies them only after the mapping dialog has actually succeeded. New regression test: `test_apply_config_mapping_cancel_leaves_window_geometry_unchanged`.

### PluginContext API Facade (#71)

Requirements: `docs/requirements/plugin-api.md` REQ-PLUGIN-060–150. This is the first piece of code in the "2.3 Plugins" epic beyond the #70 event bus, and the only thing a plugin will ever be allowed to import from the application — everything below is written with that as a hard compatibility boundary. Plan-agent reviewed before finalizing; six real gaps caught in that pass are folded into the design below rather than listed separately.

- **New top-level package `src/mdf_viewer/plugin_api/`**, a sibling of `model/`/`view/`/`view_model/`/`controller/` rather than a submodule of any of them — it's a new kind of layer (a read-only facade + registration surface for external code), not an extension of any existing one. `docs/api.md`'s module table and `CLAUDE.md`'s Project Structure listing both need this new top-level package added, in the same commit as the implementation (standard per-issue doc-sync rule), not done here.
  - `plugin_api/types.py` — frozen, read-only dataclasses: `PluginSignalView` (`metadata`, `color`, `display_mode`, `marker_shape`, `line_width`, `line_style`, `visible`, `measurement: PluginMeasurementView | None`, plus an internal `_signal_token: int` — see below), `PluginMeasurementView` (`path`, `label`, `offset_s`, `is_primary`), `PluginTabSignals` (`tab_index`, `tab_name`, `is_active`, `signals: list[PluginSignalView]`), `PluginTabCursor` (`tab_index`, `tab_name`, `is_active`, `mode: CursorMode`, `positions: list[float]`). None of these import PyQtGraph; `color: QColor` is the one Qt type carried through, matching `ActiveSignal.color`'s own existing type — same precedent `controller/interfaces.py` already sets by allowing plain Qt value types (not widgets) into otherwise-Qt-free layers.
  - `plugin_api/registry.py` — `PluginRegistry`, a plain (non-`QObject`) container holding `menu_actions: list[MenuActionRegistration]` and `dock_widgets: list[DockWidgetRegistration]`, each tagged with the owning `plugin_name`. One shared `PluginRegistry` instance is constructed once (by `app.py`) and passed into every `PluginContext` — sharing one registry across per-plugin contexts is fine precisely because every entry is `plugin_name`-tagged; #73 (not yet built) will read this registry to actually render menu items/dock widgets, #71 only populates it. `PluginRegistry.remove_registrations_for(plugin_name)` also exists now, symmetric with `PluginContext.unsubscribe_all()` below — a future plugin unload/reload (#72) needs both halves of teardown to exist from the start, not bolted on later.
  - `plugin_api/context.py` — `PluginContext(plugin_name, app: AppController, registry: PluginRegistry, tab_name_provider: Callable[[int], str] | None = None)`. One instance is constructed per plugin (at plugin-load time — #72/#74's job), so error logs and any future per-plugin permission scoping can always be attributed to a specific plugin. `tab_name_provider` defaults to `None`; when absent, tab names in the read surface fall back to `f"Tab {tab_index + 1}"` — tab display names live in `MainWindow`, not `AppController`, so this is the only source, mirroring the existing `capture_config(tab_names=...)` precedent of the view supplying names the controller doesn't own, rather than `AppController` growing new state to track them.
- **Signal handles are an opaque, controller-minted token — not `id(active_signal)`.** The MIME-drag code (`_mime.py`) already uses `id(ActiveSignal)` as a payload, but explicitly only because a drag is instantaneous and the object is guaranteed alive for its whole duration. A `PluginSignalView` can be held by a plugin indefinitely; if the original `ActiveSignal` is later garbage-collected, a *different* object can be allocated at the same address, and an `id()`-based lookup would silently return a different signal's data — wrong, not just missing. Instead, `AppController` maintains a monotonic counter minted once per `ActiveSignal` the first time it's exposed through a `PluginContext`, dropped from the lookup table at the same point `remove_signal`/`close_measurement`/tab-close teardown already removes that signal's other state (table row, cursor labels, curve). `AppController.find_active_signal_by_id(token: int) -> ActiveSignal | None` returns `None` once the token has been dropped. `PluginContext.get_samples(signal_view) -> tuple[np.ndarray, np.ndarray] | None` looks the token up and returns `None` for an already-removed signal instead of raising or returning a stale/wrong array; always returns a `.copy()` of `display_timestamps`/`data.samples`, never the live array, preserving REQ-PLUGIN-090's read-only guarantee even for the one call that does hand back real sample data.
- **`AppController.all_workspaces() -> list[TabWorkspace]` returns `list(self._workspaces)`, a shallow copy** — matching the existing `active_signals` property's own convention (`list(self.current_workspace.active)`, not the live list), not the literal live `_workspaces` reference. Needed since tab close does `del self._workspaces[index]` and reorder does `self._workspaces = [...]` reassignment; handing out the real list would let a caller observe (or be broken by) either mutation mid-iteration. `AppController.active_tab_index -> int` is a new public property (today only `_active_tab_index` private + `current_workspace` exist) so `PluginContext` can mark which `PluginTabSignals`/`PluginTabCursor` entry `is_active` without reaching into private state.
- **`PluginContext.active_signals -> list[PluginTabSignals]`**, one entry per tab in tab order, built by iterating `all_workspaces()` and wrapping each `ActiveSignal` via a `PluginSignalView.from_active()` factory (in `plugin_api/types.py`) that also mints/reuses that signal's token.
- **`PluginContext.measurements -> list[PluginMeasurementView]`**, built from `AppController.measurements` + `AppController.primary_measurement`, comparing for the Primary flag with `is` (identity), never `==` — `LoadedMeasurement` is a plain `@dataclass` with auto-generated field-equality `__eq__`, and the #106 architecture entry above already established `is`/`id()`-based comparison as the required pattern for exactly this reason (two distinct measurements can otherwise compare equal).
- **`PluginContext.cursor_positions -> list[PluginTabCursor]`**, built by iterating `all_workspaces()` and reading `ws.cursor_ctrl.snapshot()` when `cursor_ctrl` is not `None` (a tab can exist without one only in tests). `CursorController.snapshot()` returns `{"mode": str, "positions": list[float]}` — the string must be converted back with `CursorMode[snapshot()["mode"]]`, mirroring the exact lookup `CursorController.restore()` already performs; passing the raw string through as a `PluginTabCursor.mode` would be a type mismatch against REQ-PLUGIN-110.
- **`register_menu_action(label, callback)` / `register_dock_widget(title, widget_factory, mode: Literal["docked", "dialog"])`** append `MenuActionRegistration`/`DockWidgetRegistration` (tagged with `self._plugin_name`) to the shared registry — #71 only records the registration; rendering it into the real "Plugins" menu / drawer tab / dialog is #73's job (per the earlier product decision that both are stubs here). `widget_factory: Callable[[], "QWidget"]`'s `QWidget` import is `TYPE_CHECKING`-only, matching `controller/interfaces.py`'s existing convention — `plugin_api` stays Qt-widget-import-free at runtime the same way `controller/` does, even though it references the type for annotations. Each registration dataclass owns an `invoke()`/`build()` method that wraps the actual callback/factory call in `try/except Exception`, logging via `logging.getLogger("mdf_viewer.plugin_api")` and swallowing the exception — so REQ-PLUGIN-150's error-isolation guarantee is satisfied once #73 actually calls `registration.invoke()`, without #73 needing to reimplement it.
- **`PluginContext.subscribe(event_name: str, handler)` / `unsubscribe_all()`** — `subscribe` validates `event_name` against `EventBus`'s five known signal names, connects `getattr(app.events, event_name).connect(wrapped)` where `wrapped` applies the same catch/log/swallow wrapping as menu actions above, and appends `(event_name, wrapped)` to a per-context `self._subscriptions` list. `unsubscribe_all()` iterates that list and disconnects each via the exact `try: signal.disconnect(handler) except TypeError: pass` idiom already established in `plot_stripe.py` (safe no-op if already disconnected), then clears the list. Nothing calls `unsubscribe_all()` yet — that's #72's plugin-lifecycle job — but it must exist and work correctly now, same reasoning as `PluginRegistry.remove_registrations_for`.
- **Logging has no existing app-wide logger to hook into** (confirmed: no `logging.getLogger` calls anywhere in `src/`). `plugin_api` uses the stdlib `logging` module directly (`logging.getLogger("mdf_viewer.plugin_api")`) rather than inventing its own mechanism or waiting on #126 (Backlog "Feature: Logging") — stdlib logging works standalone (defaults to WARNING+ on stderr unconfigured) and #126 can attach real handlers to this same logger name later with zero `plugin_api` changes.
- **Not addressed here, by design:** actually wiring registered menu actions/dock widgets into `MainWindow` (#73), plugin loading/discovery/API version negotiation (#74), and virtual measurements for artificial-signal/custom-format plugins (#147, depends on #71 landing first).
- **Terminology drift** ("unit" meaning two different things in `plot_stripe.py`; Active Signals Table "segment" as a second name for "stripe") — resolved via the new "Glossary" section above, per the issue's own suggested fix, rather than renaming anything.

### Plugin Base Class and Lifecycle (#72)

Requirements: `docs/requirements/plugin-api.md` REQ-PLUGIN-160–190. Extends #71's already-shipped `plugin_api` package with the class a plugin author actually implements against — `PluginContext` is what a plugin *receives*, `Plugin` is what it *writes*. Plan-agent reviewed; three real gaps caught and folded in below (most importantly, a partial-activation leak of the exact shape the #71 review already warned about for long-lived handles).

- **New `plugin_api/plugin.py`**, one `Plugin` base class. `name`/`version`/`description`/`author` are plain **class-level** attributes (default `""`), not instance attributes set in `__init__` — a plugin author overrides them directly on the subclass body (REQ-PLUGIN-160).
- **`name` is validated in `__init_subclass__`, not `__init__`.** `__init_subclass__` runs at class-definition time, before any instance exists, so it can't be skipped by a plugin author who overrides `__init__` and forgets `super().__init__()` — the exact failure mode the #71-style review caught here. `context: PluginContext | None = None` and `_active: bool = False` are likewise plain class-level defaults rather than instance attributes assigned in `__init__`, so `Plugin` needs no `__init__` at all — a subclass with no constructor of its own (the expected common case) just works, and one that does define `__init__` doesn't have to remember to call `super().__init__()` for lifecycle state to exist.
- **`activate(context)` defaults to raising `NotImplementedError`** (REQ-PLUGIN-170 — mandatory, since it's the only place a plugin can ever register anything). **`deactivate()` defaults to a no-op** (REQ-PLUGIN-171).
- **Five no-op event handler methods** (`on_file_loaded`, `on_signal_added`, `on_signal_removed`, `on_selection_changed`, `on_cursor_moved`, REQ-PLUGIN-180) mirror `EventBus`'s five signal names exactly. **Auto-wiring** (REQ-PLUGIN-181) walks that fixed mapping and, for each, compares `getattr(type(self), handler_name) is not getattr(Plugin, handler_name)` — confirmed correct Python 3 semantics during review: `type(self).on_x` resolves to the plain function found via the MRO (not a bound method, so no bound-method identity instability), meaning an unoverridden handler always resolves to the exact same function object as `Plugin.on_x`, and any override — including through multiple inheritance — resolves to a distinct object. Only overridden handlers get `context.subscribe()`'d; a plugin never calls `subscribe()` itself for these five events.
- **Framework-facing lifecycle entry points, distinct names from the plugin-author-facing hooks:** `start(context) -> bool` and `stop() -> None`, called by the future loader (#74) — never called `activate`/`deactivate` directly, so the two audiences (plugin author vs. loader) can't be confused. `start()` is idempotent (returns `True` immediately if already active); `stop()` is idempotent (no-op if not active).
- **`start()`'s failure path calls `context.teardown()` before returning `False`** (Plan-review catch): if a plugin's `activate()` calls `context.register_menu_action()`/`context.subscribe()` successfully before later raising partway through, those registrations/subscriptions must not leak — since `_active` stays `False` on failure, `stop()`'s own guard would otherwise make it a permanent no-op that never cleans up the partial state, letting a dead plugin's menu action or a stray Qt connection live in the shared `PluginRegistry`/`EventBus` forever.
- **`stop()` uses try/except/log around both `deactivate()` and teardown, with `finally` blocks guaranteeing `self.context = None`/`self._active = False` unconditionally** (Plan-review catch): without this, a raising `deactivate()` or a raising teardown could leave `_active` stuck `True` with a half-torn-down plugin, and calling `stop()` again would either no-op (if the guard fires on stale `True`) or re-run `deactivate()` a second time — violating REQ-PLUGIN-171/172's "exactly once" contract. After `stop()` returns, the plugin's own bookkeeping always reflects "inactive," even on a teardown failure (accepted trade-off: the registration/subscription itself might still be leaked in that failure case, but the plugin's own state machine never desyncs from reality).
- **New `PluginContext._teardown()`** (extends #71's already-shipped `context.py`) — calls `self.unsubscribe_all()` then `self._registry.remove_registrations_for(self._plugin_name)`. Deliberately named with a leading underscore and documented as framework-internal (called only by `Plugin.start()`'s failure path and `Plugin.stop()`), not part of the plugin-author-facing contract `subscribe`/`register_menu_action`/etc. belong to — same "private by convention, used deliberately by one specific trusted caller across a module boundary" idiom this codebase already uses elsewhere (e.g. `AppController.capture_config()` reading `loader._path` directly).
- **Error isolation** (REQ-PLUGIN-190): `activate()`, `deactivate()`, and every auto-wired event handler are already covered by, respectively, `start()`'s own try/except, `stop()`'s own try/except, and `PluginContext.subscribe()`'s existing per-handler wrapping (#71) — no new wrapping mechanism needed for handlers, `Plugin` only adds the two lifecycle-call wrappers.
- **Testable without a `QApplication`**, matching the existing `CursorController`-without-`QApplication` precedent: `Plugin.start()`/`stop()` only need an object satisfying the small `subscribe`/`_teardown` surface, so unit tests can pass a lightweight stub in place of a real `PluginContext`.
- **Not addressed here, by design:** actual plugin discovery/import/instantiation and calling `start()`/`stop()` on the app's own lifecycle (#74, the plugin loader) — `Plugin` itself has no knowledge of how or when it gets constructed.
